# CLAUDE.md — Instructions for Claude Code

## Project
BSVM: An Ethereum Virtual Machine Layer 2 running on BSV.

## Specification
Read ALL files in the `spec/` directory before beginning any implementation work. The specs are numbered 00-13 and should be read in order. Each spec defines interfaces, types, and acceptance criteria for its phase. Specs 10-13 are authoritative — where they conflict with specs 00-09, specs 10-13 win.

## Key Principles

1. **Zero geth imports**: The final `go.mod` must NOT import `github.com/ethereum/go-ethereum`. All code from geth is COPIED and ADAPTED into our own packages. However, you should clone geth locally as a reference and copy code from its `core/vm/`, `core/state/`, and `trie/` packages.

2. **Interface-first**: Define interfaces before implementations. The `StateDB` interface in `pkg/vm/statedb.go` is the most important seam in the entire project.

3. **Test-driven on ethereum/tests**: After implementing the EVM extraction (Phase 1), immediately set up the ethereum/tests runner. All subsequent work should keep these tests green.

4. **Minimal dependencies**: Use only the dependencies listed in `09-IMPLEMENTATION-ORDER.md`. If you need additional libraries, prefer pure-Go implementations.

## Implementation Order

Follow the milestone order in `09-IMPLEMENTATION-ORDER.md` strictly:
0. Validation gates (Gate 0a: FRI verifier feasibility, Gate 0b: SP1 proof size)
1. `pkg/rlp/` → `pkg/types/` → `pkg/crypto/` → `pkg/vm/` (extraction) → `internal/db/` → `pkg/mpt/` → `pkg/state/`

3. `pkg/prover/` (SP1 guest program in Rust + Go host)

6. `pkg/rpc/`

Do NOT skip ahead. Each milestone builds on the previous one.
Gate 0 MUST pass before any work beyond Milestone 2.

## EVM Extraction Process

When extracting geth's `core/vm/`:

1. Clone `github.com/ethereum/go-ethereum` at the latest stable tag (v1.15.x, post-Prague, pre-EOF)
2. Copy files from `core/vm/` into `pkg/vm/`
3. For each file, replace geth import paths:
   - `github.com/ethereum/go-ethereum/common` → `github.com/icellan/bsvm/pkg/types`
   - `github.com/ethereum/go-ethereum/crypto` → `github.com/icellan/bsvm/pkg/crypto`
   - `github.com/ethereum/go-ethereum/params` → use our `pkg/vm` config types
   - `github.com/ethereum/go-ethereum/core/tracing` → `github.com/icellan/bsvm/pkg/vm/tracing`
   - `github.com/ethereum/go-ethereum/core/types` → `github.com/icellan/bsvm/pkg/types`
4. Remove any references to `state.AccessEvents`, `stateless.Witness`, Verkle-specific code
5. Keep `github.com/holiman/uint256` as-is (it's an external dependency we use directly)

## Coding Standards

- Go 1.22+
- Run `gofmt` on all files
- Run `go vet` before committing
- Every exported function has a doc comment
- Error messages are lowercase, no punctuation (Go convention)
- Use `context.Context` for cancellation in long-running operations
- Use `sync.Mutex` not `sync.RWMutex` unless read-heavy access is proven
- Prefer `uint256.Int` over `*big.Int` for EVM values (matches modern geth)
- Use structured logging (e.g., `log/slog`)

## Testing

- Write tests alongside implementation, not after
- Use table-driven tests for opcode testing
- Use `testing.Short()` to skip slow integration tests
- The ethereum/tests suite is the ultimate correctness oracle
- Fuzz critical paths: EVM execution, StateDB operations, RLP encoding

## File Naming Convention

- Types: `types.go`
- Interfaces: name after the primary interface (e.g., `statedb.go` for the StateDB interface)
- Implementations: name after the concrete type (e.g., `memory_statedb.go`)
- Tests: `*_test.go` in the same package

## Architecture Model (specs 10 + 11 + 12)

The system is an ecosystem of independent EVM instances ("shards") on BSV.
Each shard has:
- Its own covenant UTXO chain on BSV
- Its own node network (multiple nodes replicating state)
- Its own contract deployments (full EVM bytecode compatibility)
- Its own bridge covenant for BSV deposits/withdrawals

There is NO sequencer key. The STARK proof (generated by SP1 zkVM
running revm) is the sole authorization for covenant state advances.
Anyone with a valid proof can advance the state.

Optional governance keys (configurable per-shard at genesis) can
freeze/unfreeze the shard and upgrade the covenant script, but CANNOT
advance state or access bridge funds. Governance modes: none (fully
trustless), single_key, or multisig (M-of-N). See spec 12.

BSV is the consensus layer. Nodes don't run their own consensus — they
execute deterministically and race to advance the covenant on BSV.

## EVM Layer: 100% Ethereum-Compatible

The EVM layer is 100% Ethereum-compatible. Users deposit BSV via the
bridge, get wBSV, and interact via standard Ethereum tooling (MetaMask,
ethers.js, Hardhat, Foundry). Gas is paid in wBSV from L2 balance.
Users never touch BSV directly after the initial bridge deposit. All
BSV interaction (covenant advances, mining fees) is the prover's concern.

There is ONE fee model: standard Ethereum gas, paid in wBSV. No BSV-
specific gas payment modes, no direct BSV gas payments, no satoshi-
denominated gas pricing in the EVM or RPC layer.

## Packages and Concepts That Do NOT Exist

Do not create any of these:
- pkg/sequencer/ — no sequencer, no mempool, no tx ordering logic
- pkg/anchor/ — no passive anchoring service
- Any "block production timer" or "block interval" concept
- Any "sequencer key" or "operator key" that can advance state
  (governance keys can freeze/upgrade but NOT advance state)
- wBSV as a separately managed wrapped token concept (it's just the
  native EVM balance, credited by the bridge)
- Any BSV-specific gas payment logic in the EVM or RPC layer
- Direct BSV gas payment from users (Mode 1 / Mode 2 distinction)
- Fee accumulation in the covenant UTXO
- Prover reward distribution in the covenant script (prover earns via
  L2 coinbase only)

## Dual-EVM Architecture (spec 12)

The system runs TWO EVM implementations:
- Go EVM (extracted from geth): runs in the overlay node for fast
  execution. Sub-millisecond. Returns receipts immediately.
- Rust EVM (revm inside SP1 zkVM): runs inside SP1 to generate the
  STARK proof. Proves full EVM execution correctness.

Both EVMs MUST pass ethereum/tests and produce identical state roots
for identical inputs. The Go EVM is for speed. The Rust EVM inside
SP1 is for proof. If they disagree, it is a critical bug.

## Rúnar Integration

All Bitcoin Script in covenant contracts is compiled via runar-go, imported
as a Go library. Covenant contracts are in pkg/covenant/contracts/ and use
the Rúnar Go DSL. The covenant has NO sequencer signature check — only
SP1 STARK verification via a Rúnar-compiled FRI verifier using Poseidon2
and Baby Bear field arithmetic (p = 2^31 - 2^27 + 1). SP1 uses
Poseidon2 (an algebraic hash over Baby Bear) for FRI Merkle commitments.
BSV's native SHA-256 is used separately for BSVM data bindings only.

## Node Model

Each shard is run by multiple independent competing nodes:
- Nodes gossip EVM transactions to each other
- All nodes execute the same transactions deterministically
- Any node can generate a STARK proof and advance the covenant
- BSV resolves races — if two nodes advance simultaneously, one wins
- The winner earns L2 coinbase fees; losers discard their proof and
  replay the winner's batch
- No leader election, no sequencer rotation, no privileged role
- All nodes MUST verify every covenant advance by re-executing the batch
  and comparing state roots — this is a defence-in-depth check

## Execution Correctness

The STARK proof (SP1 zkVM running revm compiled to RISC-V) guarantees
FULL EVM execution correctness — every opcode, every stack operation,
every memory access, every storage write, every balance transfer, every
gas deduction. This is a complete validity proof, not a trie-integrity-
only proof.

Additionally, all nodes re-execute every batch and compare state roots
as a defence-in-depth verification. This is not the primary security
guarantee (the STARK proof is), but it provides an additional layer of
assurance.

## Speculative Receipts

In the multi-node competitive model, receipts are speculative until the
node's batch wins the covenant advance race and is proven. If the node
loses the race, some receipts may be invalidated (transactions absent
from the winning batch or now invalid against the winner's post-state).
The block tags communicate this: latest = speculative, safe = proven,
finalized = 6+ BSV confirmations. See spec 11, "Speculative Receipts."

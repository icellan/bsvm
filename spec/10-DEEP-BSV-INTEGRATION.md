# Spec 10 (Revised): Deep BSV Integration via Rúnar

## Why Rúnar, Not sCrypt

sCrypt is TypeScript-based. Rúnar fits this project differently and gives us three structural advantages:

1. **Go compiler as a library**: BSVM is written in Go and imports Rúnar's Go compiler directly as a package. The covenant scripts are compiled at build time or at genesis from Go source with no JavaScript toolchain in the critical path. Rúnar also has non-Go backends (TypeScript, Rust, Python, Zig, Ruby) but BSVM compiles exclusively through the Go backend — cross-compiler parity against the BSVM contract set is NOT a BSVM release criterion, is not tested here, and should not be relied on for auditing the covenant logic.

2. **ANF IR as the audit artifact**: Rúnar's Administrative Normal Form (ANF) intermediate representation gives every sub-expression a named temporary with canonical JSON serialization (RFC 8785). The ANF is an explicit, inspectable contract surface — auditors can read it directly rather than disassembling Script bytes. This is the audit artifact for the Go-compiled covenants.

3. **Built-in ZKP verification primitives**: Rúnar ships BN254 G1/G2/pairing, KoalaBear field arithmetic, Ext4 arithmetic, and SHA-256 Merkle primitives as built-in compiler support. The Mode 2 / Mode 3 Groth16 verifiers and the future Gate 0a Full FRI verifier compose directly from these primitives without novel Script engineering.

4. **Same author, same project**: Rúnar and bsvm share the same development context. The compiler can be evolved alongside the EVM project as new covenant requirements emerge without coordinating with an external team.

## The Covenant Contract (Rúnar Go DSL)

The state covenant is written using Rúnar's Go compiler. This contract guards the L2 state root as a stateful UTXO chain. Each L2 batch is a BSV transaction that spends the current state UTXO and creates a new one.

### The Covenant Contract

> **NOTE**: The phased covenant approach (rollup_v1.go, rollup_v2.go,
> rollup_v3.go) is **collapsed** into a single `rollup.go` from spec 12
> that has NO sequencer signature — only the STARK proof.
>
> The covenant contract has one method: `advanceState`. It takes a new state
> root, a block number, and an SP1 STARK proof. The proof is generated by
> running a Rust EVM (revm) inside the SP1 zkVM, proving full EVM execution
> correctness. The BSV covenant verifies this proof via a Rúnar-compiled FRI
> verifier using SHA256 hashing and field arithmetic over SP1's Baby Bear
> field (p = 2^31 - 2^27 + 1).

See spec 12 for the definitive `RollupContract` implementation.

### Multi-Shard Note

Each shard has its own instance of this covenant. The covenant is compiled
once by Rúnar (parameterised by the SP1 verifying key) and the compiled
Script is reused for the shard's entire lifetime. Each shard has a unique
SP1 verifying key because the chain ID is hardcoded in the guest program
for EIP-155 replay protection (see Spec 12, "Cross-shard proof replay
prevention"). The covenant Rúnar template is reusable across shards, but
the compiled output (including the embedded VK) differs per shard.

## Compilation Pipeline

Rúnar's Go compiler is imported as a library. Compilation happens at build time or genesis — no external process needed.

```go
// pkg/covenant/compile.go

func CompileCovenant(sp1VerifyingKey []byte, chainID uint64, governanceConfig GovernanceConfig) (*CompiledCovenant, error) {
    contract := contracts.RollupContract(sp1VerifyingKey, chainID, governanceConfig)

    // Go DSL → ANF IR (canonical JSON) → Bitcoin Script bytes
    anf, err := runar.ToANF(contract)
    if err != nil {
        return nil, fmt.Errorf("ANF conversion failed: %w", err)
    }
    script, err := runar.Compile(anf)
    if err != nil {
        return nil, fmt.Errorf("script compilation failed: %w", err)
    }

    return &CompiledCovenant{
        LockingScript: script,
        ANF:           anf,
        StateSize:     contract.StateSize(),
    }, nil
}
```

### ANF as the Audit Artifact

The ANF IR is the formal specification of the covenant. It is serialized as canonical JSON (RFC 8785) and published alongside the genesis. Auditors read the ANF directly to verify the covenant's semantics rather than disassembling Script bytes. Re-compiling via other Rúnar backends is NOT a supported verification path for BSVM — the Go compiler's output is the authoritative locking script.

**ANF publication**: The covenant's ANF IR is serialized as canonical JSON (RFC 8785) and published in the genesis transaction's second OP_RETURN output. Nodes joining the shard compile the same contract source through the Go compiler and verify the resulting ANF hash and script hash match the on-chain records. This ensures all nodes can independently verify the covenant logic with the same toolchain.

## Covenant State Manager

```go
// pkg/covenant/manager.go

type CovenantManager struct {
    bsvClient    BSVClient
    feeWallet    *FeeWallet       // BSV wallet for miner fee funding (NOT a sequencer key)
    covenant     *CompiledCovenant

    currentTxID  Hash
    currentVout  uint32
    currentSats  uint64
    currentState CovenantState
}

// FeeWallet manages BSV UTXOs used to pay miner fees for covenant-advance
// transactions. This is NOT a sequencer key — it has no special authority.
// The wallet only signs its own fee-funding inputs. The covenant input is
// authorized by the STARK proof, not by any signature.
type FeeWallet struct {
    key   *bsv.PrivateKey
    utxos []UTXO  // Available funding UTXOs
    mu    sync.Mutex
}

// SelectUTXO picks a UTXO large enough to cover the estimated fee.
// If no single UTXO is large enough, SelectUTXOs returns multiple
// UTXOs that sum to the required amount (the covenant-advance tx
// can have multiple fee-funding inputs, all signed by the fee wallet).
func (w *FeeWallet) SelectUTXOs(requiredSats uint64) ([]UTXO, error)

**Selection algorithm**: `SelectUTXOs` uses a deterministic greedy approach: sort UTXOs by value descending, select the fewest UTXOs whose sum >= `requiredSats + estimatedFee`. If no combination is sufficient, return `ErrInsufficientFunds`. This determinism ensures that given the same UTXO set, all nodes select the same inputs.

// Consolidate merges small UTXOs into a single larger UTXO.
// Called periodically (default: every 100 covenant advances) or when
// the UTXO count exceeds ConsolidationThreshold (default: 50).
// The consolidation transaction is a standard P2PKH spend — it has
// nothing to do with the covenant.
func (w *FeeWallet) Consolidate() (*bsv.Transaction, error) {
    w.mu.Lock()
    defer w.mu.Unlock()

    if len(w.utxos) < w.consolidationThreshold {
        return nil, nil // No consolidation needed
    }

    // Build a transaction that spends all UTXOs to a single output
    tx := bsv.NewTransaction()
    var totalSats uint64
    for _, utxo := range w.utxos {
        tx.AddInput(&bsv.TransactionInput{
            SourceTXID:       utxo.TxID,
            SourceTxOutIndex: utxo.Vout,
        })
        totalSats += utxo.Value
    }

    fee := estimateFee(tx)
    tx.AddOutput(&bsv.TransactionOutput{
        LockingScript: w.ChangeScript(),
        Satoshis:      totalSats - fee,
    })

    // Sign all inputs
    for i := range w.utxos {
        w.SignInput(tx, i, w.utxos[i])
    }

    return tx, nil
}

**Concurrency**: Consolidation and `AdvanceState` share a `sync.Mutex` (`walletMu`). A consolidation transaction is never broadcast while an `AdvanceState` transaction is in-flight. If a consolidation transaction is pending in the BSV mempool and `AdvanceState` needs a UTXO that consolidation consumed, `AdvanceState` waits for the consolidation to confirm or uses alternative UTXOs. The mutex ensures no double-spend between consolidation and covenant advances.

type FeeWalletConfig struct {
    ConsolidationThreshold int    // Default: 50 UTXOs
    ConsolidationInterval  int    // Default: every 100 advances
    MinUTXOSize           uint64  // Don't create change below this (default: 10,000 sats)
}

### Fee Wallet Bootstrap and Float Management

**Initial funding**: A new prover must fund the fee wallet with BSV before the first covenant advance. The minimum recommended float is 0.05 BSV (~50,000 satoshis), sufficient for approximately 500 covenant-advance transactions at current fee rates.

**Funding sources**: The prover funds the fee wallet by sending BSV to the wallet's P2PKH address from any external BSV wallet. This is a one-time bootstrap cost.

**Replenishment cycle**: The prover earns wBSV via L2 coinbase fees → periodically bridges wBSV to BSV via the withdrawal flow → BSV is sent to the fee wallet. This cycle has latency (withdrawal finalization takes ~6 BSV blocks), so the float must be large enough to cover operations during the replenishment window.

**Starvation behavior**: If the fee wallet balance drops below the estimated fee for the next covenant advance:
1. The node logs a critical warning: `fee wallet balance critical: X sats remaining`
2. The node continues executing blocks locally and serving RPC reads
3. The node gossips transactions to peers
4. The node does NOT attempt covenant advances until the wallet is replenished
5. Other nodes in the shard can still advance the covenant — the shard is not stalled unless ALL provers are starved

**Minimum float monitoring**: The overlay node exposes `bsv_feeWalletBalance` via RPC for monitoring.

type CovenantState struct {
    StateRoot        Hash
    BlockNumber      uint64
    // NOTE: PrevBatchHash was removed. The covenant only stores stateRoot
    // and blockNumber (see spec 12). Batch data integrity is guaranteed by
    // the batchDataHash in the STARK proof's public values (see spec 12,
    // "Batch data hash binding"). There is no need to chain batch hashes
    // in the covenant state — the STARK proof already covers this.
}

**Batch data integrity**: The STARK proof's public values include `batchDataHash = hash256(batchData)` where hash256 is double-SHA256 (`OP_HASH256`) and `batchData` is the canonical batch encoding after the 5-byte envelope header `BSVM\x02` (see Spec 11, "Canonical Batch Data Encoding Format"). The covenant binds the proof to the on-chain batch data via two checks: (1) `hash256(batchData) == batchDataHash` from the proof, verified with native `OP_HASH256`; (2) `hashOutputs` reconstruction via OP_PUSH_TX's sighash preimage to verify the OP_RETURN contains exactly this batchData. See Spec 12 for the full verification procedure.

// covenantSats is the fixed satoshi amount the covenant UTXO carries.
// It does not grow or shrink. The prover's incentive is purely L2
// coinbase fees (wBSV), not BSV from the covenant.
const covenantSats = 10000
// 10,000 satoshis — well above the standard BSV dust limit (546 sats).
// This value is fixed for the lifetime of the covenant chain. It does not
// accumulate fees — the prover's incentive is L2 coinbase fees only.

func (cm *CovenantManager) AdvanceState(
    newState  CovenantState,
    batchData []byte,
    proof     *Proof,
) (*bsv.Transaction, error) {

    tx := bsv.NewTransaction()

    // Input 0: spend current covenant UTXO — authorized by the proof,
    // NOT by a signature. The unlocking script contains the proof data
    // and new state values.
    tx.AddInput(&bsv.TransactionInput{
        SourceTXID:       cm.currentTxID,
        SourceTxOutIndex: cm.currentVout,
        UnlockingScript:  cm.buildUnlockScript(newState, proof),
    })

    // Input 1: fee-funding UTXO from the prover's BSV wallet.
    // This covers the BSV mining fee. The prover earns L2 coinbase
    // fees (wBSV) to offset this cost.
    feeUTXO, err := cm.feeWallet.SelectUTXO(estimateFee(tx))
    if err != nil {
        return nil, fmt.Errorf("no BSV available for miner fees: %w", err)
    }
    tx.AddInput(&bsv.TransactionInput{
        SourceTXID:       feeUTXO.TxID,
        SourceTxOutIndex: feeUTXO.Vout,
    })

    // Output 0: new covenant UTXO — same fixed satoshis as input.
    // The covenant script enforces this amount is preserved.
    tx.AddOutput(&bsv.TransactionOutput{
        LockingScript: cm.covenant.BuildLockingWithState(newState),
        Satoshis:      covenantSats,
    })

    // Output 1: OP_RETURN batch data
    tx.AddOutput(&bsv.TransactionOutput{
        LockingScript: opReturn(append([]byte("BSVM\x02"), batchData...)),
        Satoshis:      0,
    })

    // Output 2: prover's change from the fee-funding input
    change := feeUTXO.Value - estimateFee(tx)
    if change > dustLimit {
        tx.AddOutput(&bsv.TransactionOutput{
            LockingScript: cm.feeWallet.ChangeScript(),
            Satoshis:      change,
        })
    }

    // Sign ONLY the fee-funding input (input 1). The covenant input
    // (input 0) is authorized by the proof in the unlocking script.
    cm.feeWallet.SignInput(tx, 1, feeUTXO)

    txid, err := cm.bsvClient.Broadcast(tx)
    if err != nil {
        return nil, err
    }

    cm.currentTxID = txid
    cm.currentVout = 0
    cm.currentState = newState
    cm.feeWallet.MarkSpent(feeUTXO)
    return tx, nil
}
```

## Genesis: Creating the Initial Covenant UTXO

Genesis is the one operation that requires a funding key — to create the
initial covenant UTXO. This is NOT a sequencer key. The creator has no
ongoing authority. After genesis, the covenant is governed solely by proofs.

```go
// pkg/covenant/genesis.go

func CreateGenesisCovenant(
    bsvClient       BSVClient,
    sp1VerifyingKey []byte,
    genesisState    CovenantState,
    fundingKey      *bsv.PrivateKey, // Only used to fund genesis tx, no ongoing role
    fundingUTXO     UTXO,
    covenantSats    uint64,
) (Hash, error) {
    compiled, err := CompileCovenant(sp1VerifyingKey, chainID, governanceConfig)
    if err != nil {
        return Hash{}, err
    }

    tx := bsv.NewTransaction()
    tx.AddInput(&bsv.TransactionInput{
        SourceTXID:       fundingUTXO.TxID,
        SourceTxOutIndex: fundingUTXO.Vout,
    })
    tx.AddOutput(&bsv.TransactionOutput{
        LockingScript: compiled.BuildLockingWithState(genesisState),
        Satoshis:      covenantSats,
    })
    tx.AddOutput(&bsv.TransactionOutput{
        LockingScript: opReturn(append([]byte("BSVM\x01"), encodeGenesisConfig(genesisState)...)),
        Satoshis:      0,
    })

    // Sign the funding input (standard P2PKH). After this tx, the
    // fundingKey has no further role — the covenant is proof-governed.
    signP2PKH(tx, 0, fundingKey, fundingUTXO)
    txid, err := bsvClient.Broadcast(tx)
    if err != nil {
        return Hash{}, err
    }
    return txid, nil
}
```

### Genesis State Specification

**Initial state root**: The genesis state root is the MPT root of the genesis state, computed by:
1. Initializing an empty state trie
2. Applying all genesis allocations (predeploy contracts, prefunded accounts) as defined in the shard's genesis config
3. Computing `IntermediateRoot(true)` — the resulting 32-byte hash is the genesis state root

**Genesis block number**: Block number 0. The first executable block is block 1.

**Genesis covenant txid distribution**: The genesis covenant txid is included in the shard's bootstrap configuration file (`shard.json`). Nodes joining the shard receive this file out-of-band (e.g., downloaded from the shard operator's website, included in a Docker image, or distributed via a well-known registry). The shard config contains:
```json
{
  "shardID": "<genesis covenant txid>",
  "genesisCovenantTxID": "<txid>",
  "genesisCovenantVout": 0,
  "genesisStateRoot": "0x...",
  "sp1VerifyingKey": "0x...",
  "bootstrapPeers": ["<multiaddr>", ...]
}
```

**Shard ID**: The shard ID is the genesis covenant txid. This is globally unique (Bitcoin txids are unique) and serves as the canonical identifier for cross-shard references.

## Post-Quantum Future-Proofing

Rúnar has built-in verification for WOTS+ and SLH-DSA (FIPS 205/SPHINCS+).
Since the covenant uses STARK verification (hash-based, not EC-based),
the system is already post-quantum secure. STARKs rely on SHA256
collision resistance, not discrete log assumptions. No migration needed
for quantum resistance — this is a built-in property of the design.

## Forced Inclusion Inbox (Censorship Escape Hatch)

A separate "inbox" covenant UTXO chain enables censorship resistance.
Anyone can submit EVM transactions directly to BSV, bypassing shard nodes
entirely. Shard nodes are required to include inbox transactions within N
covenant advances — otherwise the state covenant rejects further advances.

### Inbox Covenant (Rúnar)

```go
// pkg/covenant/contracts/inbox.go

func InboxCovenant() *runar.Contract {
    c := runar.NewStatefulContract("Inbox")

    c.State("txQueueHash", runar.Bytes32) // Merkle root of queued txs
    c.State("txCount",     runar.Uint64)

    c.Method("submit", func(m *runar.MethodBuilder) {
        evmTxRLP := m.Param("evmTxRLP", runar.VarBytes)

        // Anyone can call — no signature check on the covenant.
        // The user pays the BSV mining fee for this transaction.
        txHash := m.Hash256(evmTxRLP)
        newRoot := m.MerkleAppend(c.GetState("txQueueHash"), txHash)

        c.SetState("txQueueHash", newRoot)
        c.SetState("txCount", m.Add(c.GetState("txCount"), runar.Uint64Literal(1)))

        m.RequireStateOutput()

        // Store the full tx in OP_RETURN for data availability
        m.RequireOutputScript(1, m.OpReturnPrefix(
            m.Cat([]byte("BSVM\x11"), evmTxRLP),
        ))
    })

    return c.Build()
}
```

### State Covenant Integration (via STARK Public Values)

The inbox integration is handled through STARK public values, NOT
through cross-covenant Script introspection. The SP1 guest program
processes inbox transactions and commits the inbox state as public
values. The covenant reads these values and enforces inclusion rules.
This keeps the covenant script simple (a few comparisons and counter
logic) while moving the complex inbox verification into the STARK
proof where it's free.

**Covenant state field**: `advancesSinceInbox` (`uint64`), added
alongside `stateRoot` and `blockNumber`. See the covenant contract
in Spec 12 for the full state declaration.

**Public values**: The SP1 guest commits two inbox-related values
(see Spec 12, "Proof Public Values Layout"):
- `inboxRootBefore` (offset 176, 32 bytes): The inbox hash chain
  root before this batch. `bytes32(0)` if inbox is empty or disabled.
- `inboxRootAfter` (offset 208, 32 bytes): The inbox hash chain root
  after this batch. `bytes32(0)` if all inbox txs were included
  (queue fully drained). Equals `inboxRootBefore` if no inbox txs
  were included in this batch.

**How it works**:

1. The overlay node reads the inbox covenant's UTXO chain on BSV to
   determine the current inbox root and pending inbox transactions.
2. The overlay node passes the inbox transactions and inbox root to
   the SP1 guest as host inputs.
3. The SP1 guest:
   a. Verifies `inboxRootBefore` matches the hash chain of the provided
      inbox transactions (recomputes: `root = SHA256(SHA256(prev || txHash))`
      for each tx — a simple hash chain, not a Merkle tree).
   b. Executes the inbox transactions through revm (they are EVM txs).
   c. Computes `inboxRootAfter` (the root after removing included txs
      from the queue, or `bytes32(0)` if all were included).
   d. Commits both values as public outputs.
4. The covenant extracts `inboxRootBefore` and `inboxRootAfter` from
   the STARK proof's public values and enforces the inclusion rule:

```go
// Added to advanceState in rollup.go (Spec 12 covenant contract)

// Inbox forced inclusion parameters (compile-time constant)
const maxInboxAdvances = 10

// Extract inbox roots from STARK public values
inboxRootBefore := m.ExtractBytes32(publicValues, 176)
inboxRootAfter  := m.ExtractBytes32(publicValues, 208)

// Determine if inbox txs were included in this batch:
// If inboxRootAfter != inboxRootBefore, the guest processed inbox txs.
// If inboxRootAfter == bytes32(0), the queue was fully drained.
inboxIncluded := m.Not(m.Equal(inboxRootBefore, inboxRootAfter))

// Increment the advances-since-inbox counter
newAdvanceCount := m.Add(c.GetState("advancesSinceInbox"), runar.Uint64Literal(1))

// Enforce: if counter exceeds threshold AND inbox has pending txs,
// the batch MUST include inbox txs. Three cases pass:
//   1. Counter is below threshold (we're not overdue)
//   2. Inbox is empty (inboxRootBefore == 0, nothing to include)
//   3. Inbox txs were included AND all were processed (after == 0)
//      OR inbox txs were included (partial drain allowed if counter
//      resets — but we require full drain when forced)
m.Require(
    m.Or(
        m.LessThan(newAdvanceCount, runar.Uint64Literal(maxInboxAdvances)),
        m.Equal(inboxRootBefore, runar.Bytes32Zero), // Inbox empty
        m.Equal(inboxRootAfter, runar.Bytes32Zero),  // All inbox txs included
    ),
    "forced inclusion: inbox txs must be fully included within 10 advances",
)

// Reset counter if inbox txs were included, otherwise increment
m.IfElse(inboxIncluded,
    func() { c.SetState("advancesSinceInbox", runar.Uint64Literal(0)) },
    func() { c.SetState("advancesSinceInbox", newAdvanceCount) },
)
```

### Why STARK Public Values Instead of Cross-Covenant Introspection

The previous design required the state covenant to read the inbox
covenant's state from a different BSV transaction. This required:
- A new calling convention for cross-covenant output references
- Merkle proof verification in the covenant script (adding ~20KB)
- The unlocking script to include serialized output data from
  the inbox covenant transaction
- Verification that the referenced output matches the inbox script hash

All of this complexity is eliminated by moving inbox verification
into the STARK proof. The SP1 guest already runs arbitrary Rust code
— verifying a hash chain of inbox transactions costs ~100 SHA256
operations, which is negligible inside the zkVM. The covenant only
needs to read two 32-byte values from the proof and compare them
against bytes32(0). This adds ~10 opcodes to the covenant script
instead of ~20KB.

### Inbox Age (Time-Based Fallback)

The advance-count threshold (10 advances) prevents censorship during
active periods. For idle periods when no covenant advances are
happening, a time-based fallback using `nLockTime` /
`OP_CHECKLOCKTIMEVERIFY` checks BSV block height:

- If `advancesSinceInbox >= maxInboxAdvances` AND the current BSV
  block height is more than `maxInboxBSVBlocks` (default: 6, ~1 hour)
  ahead of the last advance, the inbox must be drained.

The time-based check is enforced in the covenant via the sighash
preimage's `nLockTime` field. This ensures that even a completely
idle shard cannot censor inbox transactions for more than ~1 hour.

### Overlay Node Responsibilities

The overlay node's `InboxMonitor` (analogous to `BridgeMonitor`):

1. Watches the inbox covenant UTXO chain on BSV for new `submit` txs
2. Extracts the EVM transaction RLP from each inbox tx's OP_RETURN
3. Maintains a local queue of pending inbox transactions
4. Computes the current inbox root (hash chain of pending tx hashes)
5. When preparing a batch, passes the inbox queue to the SP1 guest
6. After a successful advance, removes included inbox txs from the queue

```go
// pkg/overlay/inbox_monitor.go

type InboxMonitor struct {
    bsvClient     BSVClient
    inboxTxID     Hash    // Current inbox covenant UTXO
    pendingQueue  [][]byte // Pending EVM tx RLP bytes
    inboxRoot     Hash     // Current hash chain root
}

func (m *InboxMonitor) CurrentRoot() Hash { return m.inboxRoot }
func (m *InboxMonitor) PendingTxs() [][]byte { return m.pendingQueue }
```

## Covenant Migration

If a critical bug is discovered in the guest program or covenant logic, a migration transfers state to a new covenant with an updated verifying key.

### Migration Procedure

1. **Build new guest program**: Fix the bug, compile to new SP1 ELF, derive new verifying key `VK_new`
2. **Compile new covenant**: `CompileCovenant(VK_new)` produces new locking script
3. **Generate migration proof**: The migration proof proves:
   - The current state root is valid (standard state transition proof)
   - The new covenant script hash is committed as a public value
   - The migration is authorized by the current covenant's `migrate` method
4. **Broadcast migration transaction**: Spends the current covenant UTXO → creates new covenant UTXO with the new locking script and the same state root + block number
5. **Nodes detect migration**: Overlay nodes monitoring the covenant chain see the output script change, download the new ANF from the migration tx's OP_RETURN, verify the ANF compiles to the new script, and update their local covenant reference

### Migration Method in Covenant

```go
func (c *StateCovenant) Migrate(m *runar.Method) {
    proof := m.Param("proof", runar.Bytes)
    newScript := m.Param("newScript", runar.Bytes)
    batchData := m.Param("batchData", runar.Bytes)
    
    // Verify the STARK proof (same as advanceState)
    m.Verify(c.sp1VK, proof)
    
    // The proof's public values commit to the new script hash
    pubVals := m.ExtractPublicValues(proof)
    newScriptHash := m.ExtractBytes32(pubVals, 240) // offset after standard fields (see spec 12, Public Values Layout)
    m.RequireEqual(m.SHA256(newScript), newScriptHash)
    
    // Output 0: new covenant with updated script, same state
    m.RequireOutputScript(0, newScript)
    m.RequireOutputValue(0, c.covenantSats)
    
    // Output 1: batch data OP_RETURN (includes new ANF IR)
    m.RequireOutputScript(1, m.OpReturnPrefix([]byte("BSVM\\x02")))
}
```

### Limitations

- Migration requires a valid STARK proof — it cannot be done via admin key or multisig
- The guest program must be updated to include a `migrate` flag that commits the new script hash as a public value
- Migration is a one-time, non-reversible covenant UTXO spend — the old covenant is consumed
- All nodes must update to software that recognizes the new verifying key

## Summary: The Full Token-Spend Story

| Action | BSV Spend | Mechanism |
|--------|-----------|-----------|
| EVM call | wBSV burned as gas on L2 (backed 1:1 by locked BSV) | Prover collects L2 coinbase fees, pays BSV miner fees from own wallet |
| L2 block production | **Covenant UTXO spent + recreated** (fixed sats) with new state root | SP1 STARK proof verified via Rúnar-compiled FRI verifier (SHA256 + Baby Bear field ops) |
| Bridge deposit | **BSV locked** in bridge covenant UTXO | Rúnar bridge contract |
| Bridge withdrawal | **BSV released** from bridge covenant after proof verification | Covenant-enforced release |

The L2 is not floating above BSV — it is threaded through BSV's UTXO set. The covenant UTXO IS the L2's heartbeat. Every block is a BSV transaction. Every state root is guarded by Bitcoin Script compiled by Rúnar.

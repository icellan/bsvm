# Spec 12: Full EVM Validity Proofs via SP1 zkVM

## Objective

Every covenant advance requires a STARK proof that the EVM state transition
is correct. Not trie integrity. Not Merkle consistency. Full EVM execution
correctness — every opcode, every stack operation, every memory access,
every storage write, every balance transfer, every gas deduction. Proven.

The proof is generated by running a Rust EVM (revm) inside the SP1 zkVM,
which produces a STARK proof of correct RISC-V execution. The BSV covenant
verifies this proof via a Rúnar-compiled FRI verifier. If the proof is
invalid, BSV miners reject the transaction. There is no trusted sequencer,
no optimistic assumption, no fraud window. The math is the security.

---

## Why SP1, Not a Custom Circuit

Building a custom zkEVM circuit is a 2-4 year, $50-100M effort. Polygon,
Scroll, zkSync, and Linea each built theirs with teams of 20+ cryptographic
engineers. We are not going to replicate this.

SP1 is an open-source STARK-based zkVM that proves the correct execution
of arbitrary Rust programs compiled to RISC-V. It is audited by four
firms (Veridise, Cantina, Zellic, KALOS), production-ready, and secures
over $1B in TVL across production L2s. SP1 Reth — a full Ethereum block
prover — was built in ~2,000 lines of Rust.

SP1 includes precompiles (optimised circuits) for keccak256 and secp256k1,
which are the two most expensive EVM operations. These reduce proving cost
by 5-10× compared to naive RISC-V execution of these operations.

We use SP1 to prove the execution of revm (the standard Rust EVM, used
by Reth and dozens of production systems). The proof attests: "this EVM
interpreter ran these transactions against this pre-state and produced
this post-state." Every opcode is covered. Every state change is proven.

### SP1 Version Requirements

**Pinned version**: SP1 v4.1.1 (`sp1-zkvm = "4.1.1"` in guest `Cargo.toml`).
This is the exact version all implementations must use. The SP1 proof
format, verifying key derivation, and precompile behavior are
version-dependent — even minor version bumps can change the VK.

```toml
# prover/guest/Cargo.toml
[dependencies]
sp1-zkvm = "=4.1.1"
```

If a newer SP1 version is needed (e.g., for performance or bug fixes),
follow the version upgrade policy below.

**Version upgrade policy**: SP1 version upgrades require:
1. Re-derive the verifying key from the updated guest ELF
2. If the VK changes: covenant migration (see Spec 10, Covenant Migration)
3. If the VK is unchanged (compatible upgrade): deploy updated prover software, no covenant change needed

**Proof format**: SP1 proofs are serialized using SP1's native `bincode`
format. The covenant's FRI verifier operates on the raw proof field
elements, not the serialized wrapper.

### SP1 Proof Binary Layout

The exact byte-level layout of an SP1 proof must be documented during
Gate 0b (spec 09, Milestone 0, step 2) by tracing through SP1 v4.1.1
source code. The high-level structure is:

```
SP1ProofWithPublicValues:
├── proof: SP1Proof
│   └── ShardProofs[]
│       ├── commitment (PcsProverData)
│       │   ├── main_commit (Merkle root, 32 bytes)
│       │   ├── permutation_commit (Merkle root, 32 bytes)
│       │   └── quotient_commit (Merkle root, 32 bytes)
│       ├── opening (PcsOpeningProof)
│       │   ├── fri_proof (FriProof)
│       │   │   ├── commit_phase_commits (Vec<Merkle roots>)
│       │   │   ├── query_proofs (Vec<QueryProof>)
│       │   │   │   ├── commit_phase_openings (field element evaluations)
│       │   │   │   └── decommitments (Merkle authentication paths)
│       │   │   └── final_poly (constant polynomial value)
│       │   └── rounds (evaluation points and values)
│       └── public_values (Vec<BabyBear>)
├── public_values: Vec<u8> (the committed public outputs)
└── sp1_version: String
```

Key SP1 source files for documenting the layout:

- `sp1-stark/src/verifier.rs` — verification logic
- `sp1-core-machine/src/utils/prove.rs` — proof generation
- `sp1-recursion/core/src/stark/` — recursive proof structures

### Two EVMs, One Truth

The system runs two EVM implementations that must produce identical results:

| | Go EVM (geth extraction) | Rust EVM (revm inside SP1) |
|---|---|---|
| Purpose | Fast local execution | Proven execution |
| Speed | Sub-millisecond | Seconds (proving) |
| Output | State root + receipts | State root + STARK proof |
| When | Immediately on tx arrival | Async, batched |
| Where | Inside overlay node process | External SP1 prover process |

Both pass ethereum/tests. Both produce the same state root for the same
inputs. This is guaranteed by the Ethereum test suite — it is the
canonical correctness oracle for any EVM implementation. If both EVMs
pass the same tests, they agree on every input.

The Go EVM is for user-facing speed. The Rust EVM inside SP1 is for
cryptographic proof. The user gets their receipt in milliseconds. The
proof follows in seconds. The BSV covenant only accepts the proof.

## Component 1: SP1 Guest Program (Rust)

The guest program runs inside SP1's RISC-V zkVM. It is a self-contained
Rust program that:

1. Reads inputs: pre-state root, serialised state, transaction batch
2. Loads the state into revm
3. Executes each transaction through revm
4. Outputs: post-state root, receipt hashes, gas used

SP1 proves that this program executed correctly on these inputs and
produced these outputs. That proof covers every EVM opcode that revm
executed.

```
prover/guest/
├── Cargo.toml          # SP1 guest dependencies (revm, alloy, sp1-zkvm, eth-trie)
└── src/
    ��── main.rs         # Guest program (~2,000 lines)
    └── mpt.rs          # Ethereum MPT wrapper (~500 lines)
```

### Rust MPT Library for State Root Computation

The SP1 guest must compute Ethereum-compatible Merkle Patricia Trie
(MPT) state roots. revm's `InMemoryDB` does NOT produce MPT roots —
it is a flat key-value store. The guest needs a full MPT implementation
in Rust to:

1. **Verify the pre-state root**: Load account and storage data from
   the Merkle proofs in the state export. Verify each proof against the
   claimed pre-state root. This ensures the Go host cannot lie about
   the pre-state.

2. **Compute the post-state root**: After revm executes all transactions,
   apply the state diffs (modified accounts and storage slots) to the
   MPT and recompute the root hash.

**Library**: Use `citrea-primitives/eth-trie` (a Rust implementation of
Ethereum's Merkle Patricia Trie used by Citrea's SP1-based rollup) or
`alloy-trie` (from the Alloy ecosystem). Both produce roots identical
to geth's MPT. The choice depends on SP1 compatibility — the library
must compile to RISC-V and not use features incompatible with SP1's
zkVM (e.g., no filesystem access, no threading, no randomness).

SP1's built-in keccak256 precompile (~100K RISC-V cycles per hash vs
~500K for naive implementation) makes Keccak256-based MPT efficient
to prove. No Poseidon alternative is needed.

**The `EthMPT` wrapper** in `mpt.rs` provides:

**Cycle cost estimate**: MPT operations are dominated by keccak256
hashing. For a 128-tx batch touching ~500 unique storage slots:
- ~1,000 keccak256 calls for Merkle proof verification

### Guest program structure (conceptual)

```rust
// prover/guest/src/main.rs

#![no_main]
sp1_zkvm::entrypoint!(main);

use revm::{Evm, db::InMemoryDB};

// Chain ID is a compile-time constant, set per shard at build time.
// This ensures the guest ELF (and therefore the SP1 verifying key)
// is unique per shard. See "Cross-shard proof replay prevention".
const CHAIN_ID: u64 = 8453111; // Set during `cargo prove build` per shard

pub fn main() {
    // 1. Read inputs from the SP1 host
    let pre_state_root = sp1_zkvm::io::read::<[u8; 32]>();
    let serialised_state = sp1_zkvm::io::read::<Vec<u8>>();
    let transactions = sp1_zkvm::io::read::<Vec<Vec<u8>>>();
    let block_context = sp1_zkvm::io::read::<BlockContext>();

    // 2. Deserialise state into revm's in-memory DB and the MPT.
    // The serialised_state includes Merkle proofs for each account
    // and storage slot (see StateExport type in state_export.go).
    // We load data into both revm's DB (for execution) and the MPT
    // (for state root computation).
    let mut db = InMemoryDB::new();
    let mut mpt = EthMPT::new();  // Ethereum-compatible Merkle Patricia Trie
    load_state_with_proofs(&mut db, &mut mpt, &serialised_state);

    // 3. Verify the pre-state root matches.
    // The MPT was populated from the Merkle proofs in the state export.
    // This verification ensures the host did not lie about the pre-state.
    let computed_pre_root = mpt.root_hash();
    assert_eq!(computed_pre_root, pre_state_root, "pre-state root mismatch");

    // 4. Execute each transaction through revm
    let mut receipts = Vec::new();
    let mut gas_used_total = 0u64;

    for tx_bytes in &transactions {
        let tx = decode_transaction(tx_bytes);

        let mut evm = Evm::builder()
            .with_db(&mut db)
            .with_block_env(block_context.to_block_env())
            .with_tx_env(tx.to_tx_env())
            .build();

        let result = evm.transact().unwrap();

        // Apply state changes
        evm.db().commit(result.state);

        // Collect receipt
        let receipt = build_receipt(&result, gas_used_total);
        gas_used_total += result.gas_used();
        receipts.push(receipt);
    }

    // 5. Compute post-state root.
    // Apply all state changes from revm execution to the MPT, then
    // recompute the root hash. This produces an Ethereum-compatible
    // Keccak256 MPT root that matches the Go EVM's output.
    mpt.apply_state_changes(&db);
    let post_state_root = mpt.root_hash();

    // 6. Detect withdrawals and build withdrawal Merkle tree.
    // Scan receipts for WithdrawalInitiated events from the bridge contract.
    // For each withdrawal, read the withdrawalHashes[nonce] value from
    // the bridge contract's storage (it was written during EVM execution).
    // Build a binary SHA256 Merkle tree of all withdrawal hashes.
    // This tree uses SHA256 (not Keccak) so the bridge covenant can
    // verify inclusion using native BSV opcodes — no Keccak in Script.
    let mut withdrawal_hashes: Vec<[u8; 32]> = Vec::new();
    for receipt in &receipts {
        for log in &receipt.logs {
            if log.address == BRIDGE_CONTRACT_ADDRESS
                && log.topics[0] == WITHDRAWAL_INITIATED_TOPIC
            {
                let nonce = decode_uint256(&log.topics[3]); // indexed nonce
                // Read the pre-computed hash256 from contract storage.
                // The L2Bridge contract stores: withdrawalHashes[nonce] =
                //   hash256(bsvAddress || satoshiAmount_u64_be || nonce_u64_be)
                let slot = storage_slot_for_mapping(WITHDRAWAL_HASHES_SLOT, nonce);
                let hash = db.storage(BRIDGE_CONTRACT_ADDRESS, slot);
                withdrawal_hashes.push(hash.into());
            }
        }
    }
    let withdrawal_root = sha256_merkle_root(&withdrawal_hashes);
    // withdrawal_root is bytes32(0) if no withdrawals in this batch.

    // 7. Compute batch data hash for data availability binding.
    // This binds the STARK proof to the specific batch data published in
    // the OP_RETURN output, preventing a malicious prover from posting a
    // valid proof with garbage batch data (which would break replay
    // verification by other nodes).
    //
    // CRITICAL: Uses hash256 (double-SHA256), NOT keccak256. The BSV
    // covenant must independently verify this hash using native OP_HASH256.
    // Keccak256 is not available in Bitcoin Script. Without this binding,
    // a malicious prover could post a valid proof for batch B1 alongside
    // different batch data B2 in the OP_RETURN — nodes replaying from
    // BSV data would execute B2 and compute wrong state roots.
    let batch_data = encode_batch_for_da(&transactions, &block_context);
    let batch_data_hash = hash256(&batch_data); // SHA256(SHA256(data))

    // 8. Inbox integration (see Spec 10, "Inbox Integration via STARK Public Values").
    //    The host provides the inbox state; the guest verifies and commits it.
    let inbox_root_before = sp1_zkvm::io::read::<[u8; 32]>(); // from host
    let inbox_txs = sp1_zkvm::io::read::<Vec<Vec<u8>>>();      // inbox txs to include
    let inbox_root_after = process_inbox_txs(
        &mut db, &mut mpt, &inbox_root_before, &inbox_txs, &mut receipts, &mut gas_used_total,
    );
    // inbox_root_after is bytes32(0) if all inbox txs were included (queue drained),
    // equals inbox_root_before if no inbox txs were included in this batch.

    // 9. Migration script hash (bytes32(0) for normal advances, set by
    //    migrate path only — not implemented in the standard guest flow).
    let migration_script_hash: [u8; 32] = [0u8; 32]; // No migration in this advance

    // 10. Commit public outputs (these are visible to the verifier).
    //     ORDER MUST MATCH the Public Values Layout table exactly.
    sp1_zkvm::io::commit(&pre_state_root);        // offset 0
    sp1_zkvm::io::commit(&post_state_root);        // offset 32
    sp1_zkvm::io::commit(&keccak256(&encode_receipts(&receipts))); // offset 64
    sp1_zkvm::io::commit(&gas_used_total);         // offset 96
    sp1_zkvm::io::commit(&batch_data_hash);        // offset 104  [hash256, NOT keccak]
    sp1_zkvm::io::commit(&CHAIN_ID.to_be_bytes()); // offset 136
    sp1_zkvm::io::commit(&withdrawal_root);        // offset 144
    sp1_zkvm::io::commit(&inbox_root_before);      // offset 176
    sp1_zkvm::io::commit(&inbox_root_after);       // offset 208
    sp1_zkvm::io::commit(&migration_script_hash);  // offset 240
}
```

### Deposit System Transaction Handling in SP1 Guest

Deposit transactions are identified by a reserved `txType` byte (0x7E) in the RLP-encoded transaction. The guest handles them as direct state mutations — NO EVM calls, NO Solidity execution:

```rust
// Bridge contract address and totalDeposited storage slot.
// These are compile-time constants matching the Go block executor.
const BRIDGE_CONTRACT: Address = address!("4200000000000000000000000000000000000010");
const TOTAL_DEPOSITED_SLOT: U256 = U256::from(4); // Solidity storage slot 4

for tx in transactions {
    if tx.tx_type() == 0x7E {
        // Deposit: direct state mutations, NOT an EVM call
        let deposit = decode_deposit_tx(&tx);

        // 1. Credit recipient balance directly
        state.add_balance(deposit.to, deposit.value);

        // 2. Update bridge contract's totalDeposited via direct storage
        //    mutation — identical to the Go block executor's approach.
        //    No Solidity code is executed. No recordDeposit() call.
        let current_total = db.storage(BRIDGE_CONTRACT, TOTAL_DEPOSITED_SLOT);
        let new_total = current_total + deposit.value;
        db.insert_account_storage(BRIDGE_CONTRACT, TOTAL_DEPOSITED_SLOT, new_total);

        // Deposit does not consume gas, does not increment nonce
        // Receipt: status=1, gasUsed=0, logs=[]
        receipts.push(deposit_receipt(deposit));
    } else {
        // Standard EVM transaction
        let result = evm.execute_tx(&tx);
        receipts.push(result.receipt);
    }
}
```

The deposit `txType` 0x7E is a reserved type not used by any Ethereum EIP. The Go overlay node uses the same type byte when encoding deposits into batch data. Both the Go block executor and the Rust SP1 guest perform identical state mutations: `AddBalance` to the recipient and a direct storage write to `totalDeposited` at slot 4. No Solidity code is executed in either path.

### What SP1 proves

The STARK proof attests exactly this:

> "The RISC-V program (compiled from the above Rust code, which runs revm)
> was executed on the given inputs. The execution consumed X cycles. The
> committed public outputs are: pre_state_root, post_state_root,
> receipts_hash, gas_used, batch_data_hash, chain_id, withdrawal_root,
> migrate_script_hash."

This covers:
- Every EVM opcode executed by revm (ADD, MUL, SLOAD, SSTORE, CALL, ...)
- Every stack push/pop
- Every memory read/write
- Every storage read/write
- Every balance transfer
- Every gas deduction
- Every contract creation and self-destruct
- Every log emission
- Every precompile invocation
- Correct nonce management
- Correct sender recovery from signatures
- Correct contract address derivation

There is no gap. The proof covers the entire EVM execution. A malicious
prover cannot claim Alice sent tokens to the attacker when she sent them
to Bob — the RISC-V trace would not match, and the proof would not verify.

### Guest Program Error Handling

The SP1 guest program must handle errors gracefully — a panic produces no proof and stalls the pipeline.

| Error | Guest Behavior | Host Recovery |
|-------|---------------|---------------|
| Pre-state root mismatch | Commit error code 0x01 as public value, exit cleanly | Host detects error code, re-exports state, retries |
| Transaction decode failure | Skip malformed tx, continue executing remaining txs | Host logs warning, investigates batch encoding |
| Out of memory (state too large) | Exit with SP1 system error | Host splits batch into smaller sub-batches |
| Storage slot not in export | Commit error code 0x02, exit cleanly | Host re-records access, re-exports with full coverage |
| revm internal error | Commit error code 0xFF, exit cleanly | Host enters diagnostic mode (EVM disagreement protocol) |

**Error code encoding**: If the guest encounters an error, it commits a 1-byte error code at public values offset 0 (replacing the first byte of preStateRoot). The host checks this byte before using the proof. A non-zero first byte means the proof is invalid and should not be broadcast.

## Component 2: SP1 Host / Prover Integration (Go)

The overlay node (Go) invokes the SP1 prover for each batch. The prover
is a Rust binary that the overlay node calls as a subprocess, or a remote
prover service.

```go
// pkg/prover/host.go

// SP1Prover invokes the SP1 proving system to generate a STARK proof
// of correct EVM execution.
type SP1Prover struct {
    guestELF    []byte          // Compiled SP1 guest program (RISC-V ELF)
    proverMode  ProverMode      // Local, Network, or Mock
    config      SP1ProverConfig
}

type ProverMode int
const (
    ProverLocal   ProverMode = iota // Run SP1 prover locally
    ProverNetwork                   // Use SP1 prover network (remote)
    ProverMock                      // Skip proving (testing only)
)

type SP1ProverConfig struct {
    GuestELFPath    string        // Path to compiled guest ELF
    ProverBinary    string        // Path to SP1 prover binary (local mode)
    ProverNetworkURL string       // SP1 prover network URL (network mode)
    Timeout         time.Duration // Max proving time before abort
    Workers         int           // Parallel proving workers
}

// ProveInput contains everything the SP1 guest program needs.
type ProveInput struct {
    PreStateRoot    Hash
    SerialisedState []byte   // State snapshot for revm to load
    Transactions    [][]byte // RLP-encoded EVM transactions
    BlockContext    BlockContext
}

// ProveOutput is the result of proving.
type ProveOutput struct {
    PostStateRoot Hash
    ReceiptsHash  Hash
    GasUsed       uint64
    Proof         []byte   // SP1 STARK proof (serialised)
    PublicValues  []byte   // Committed public outputs
    ProvingTime   time.Duration
}

// Prove generates a STARK proof of correct EVM execution.
func (p *SP1Prover) Prove(input *ProveInput) (*ProveOutput, error) {
    // 1. Serialise inputs for the SP1 guest
    stdin := serialiseProveInput(input)

    // 2. Invoke SP1 prover
    switch p.proverMode {
    case ProverLocal:
        return p.proveLocal(stdin)
    case ProverNetwork:
        return p.proveNetwork(stdin)
    case ProverMock:
        return p.proveMock(stdin)
    }
}

func (p *SP1Prover) proveLocal(stdin []byte) (*ProveOutput, error) {
```

### Host-Guest Communication Format

The Go host and Rust SP1 guest communicate via SP1's standard
`sp1_zkvm::io::read` / `sp1_zkvm::io::commit` mechanism. The host
writes inputs to the SP1 prover's stdin using SP1's `SP1Stdin` type,
which serializes values via `bincode`. The guest reads them in the
same order.

**Host → Guest inputs** (written in this order via `SP1Stdin::write`):

| # | Type | Size | Description |
|---|------|------|-------------|
| 1 | `[u8; 32]` | 32 bytes | Pre-state root |
| 2 | `Vec<u8>` | variable | State export (zstd-compressed RLP, see "State Export Serialization Format" below) |
| 3 | `Vec<Vec<u8>>` | variable | Transaction batch (each element is one EIP-2718 typed tx envelope) |
| 4 | `BlockContext` | ~100 bytes | Block context (timestamp, block number, coinbase, prevrandao, etc.) |
| 5 | `[u8; 32]` | 32 bytes | Inbox root before (bytes32(0) if inbox disabled) |
| 6 | `Vec<Vec<u8>>` | variable | Inbox transactions (empty vec if none) |

All types are serialized via `bincode` (SP1's default). `Vec<u8>` is
length-prefixed (u64 LE length, then raw bytes). `Vec<Vec<u8>>` is
doubly length-prefixed.

**Guest → Host outputs** (committed via `sp1_zkvm::io::commit`):

The guest commits the public values in the exact order specified in
the "Proof Public Values Layout" table (272 bytes total). These are
extracted from the proof by the covenant at fixed byte offsets.

**Go host implementation**: The Go host builds the stdin buffer by
serializing each input using the same `bincode` wire format that the
Rust SP1 SDK uses. This is implemented in `pkg/prover/host.go` via
the `serialiseProveInput` function, which must produce byte-identical
output to the Rust `SP1Stdin::write` calls.

The simplest implementation strategy: write a thin Rust binary
(`prover/host-bridge/`) that accepts the Go host's inputs as JSON on
stdin, converts them to `SP1Stdin`, invokes the SP1 prover, and
returns the proof + public values as JSON on stdout. This avoids
reimplementing `bincode` serialization in Go.

```
Go host  →  JSON stdin  →  Rust host-bridge  →  SP1Stdin  →  SP1 prover
                                                              ↓
Go host  ←  JSON stdout ←  Rust host-bridge  ←  proof+PV  ←  SP1 prover
```

### State Serialisation

The SP1 guest needs the account/storage state to load into revm. We
serialise only the state that the batch will access (the "access set"):

```go
// pkg/prover/state_export.go

// StateExport contains the subset of state needed by the SP1 guest,
// including Merkle proofs so the guest can verify the pre-state root
// without trusting the host.
type StateExport struct {
    PreStateRoot Hash
    Accounts     []AccountProof
}

type AccountProof struct {
    Address      Address
    Nonce        uint64
    Balance      *uint256.Int
    CodeHash     Hash
    StorageRoot  Hash
    AccountProof [][]byte     // Merkle proof: state root → account node
    Code         []byte       // Full bytecode (if contract)
    StorageProofs []StorageProof
}

type StorageProof struct {
    Key   Hash
    Value Hash
    Proof [][]byte // Merkle proof: storage root → slot value
}

// ExportStateForProving serialises the accessed state with Merkle proofs.
//
// IMPORTANT: The statedb parameter MUST be opened at the PRE-STATE root
// (before execution), not the post-execution statedb. The Merkle proofs
// must prove account/storage values in the pre-state trie so the SP1
// guest can verify it starts from the correct state root. The caller
// opens a fresh read-only statedb at the parent block's state root:
//
//   preStateDB, _ := state.New(parentHeader.StateRoot, db)
//   export, _ := ExportStateForProving(preStateDB, accounts, slots)
//
// The SP1 guest verifies the pre-state root by checking each account's
// Merkle proof against PreStateRoot, then checking each storage slot's
// Merkle proof against the account's StorageRoot. This ensures the host
// cannot lie about the pre-state.
func ExportStateForProving(statedb *state.StateDB, accessedAccounts []Address, accessedSlots map[Address][]Hash) ([]byte, error) {
    var export StateExport
    export.PreStateRoot = statedb.IntermediateRoot(false)
    
    for _, addr := range accessedAccounts {
        // Generate Merkle proof for this account in the state trie
        accountProof, err := statedb.GetProof(addr)
        if err != nil {
            return nil, fmt.Errorf("account proof for %s: %w", addr, err)
        }
        
        acc := AccountProof{
            Address:      addr,
            Nonce:        statedb.GetNonce(addr),
            Balance:      statedb.GetBalance(addr),
            CodeHash:     statedb.GetCodeHash(addr),
            StorageRoot:  statedb.GetStorageRoot(addr),
            AccountProof: accountProof,
            Code:         statedb.GetCode(addr),
        }
        
        for _, slot := range accessedSlots[addr] {
            // Generate Merkle proof for this storage slot
            storageProof, err := statedb.GetStorageProof(addr, slot)
            if err != nil {
                return nil, fmt.Errorf("storage proof for %s/%s: %w", addr, slot, err)
            }
            acc.StorageProofs = append(acc.StorageProofs, StorageProof{
                Key:   slot,
                Value: statedb.GetState(addr, slot),
                Proof: storageProof,
            })
        }
        export.Accounts = append(export.Accounts, acc)
    }
    
    return rlp.Encode(export)
}
```

The Go EVM executes first and records which accounts/slots are accessed
(via the EVM's access list tracking). Only those accounts/slots are
exported, along with their Merkle proofs. The SP1 guest verifies each
Merkle proof against the pre-state root, so the overlay node cannot lie
about the state.

If the SP1 guest accesses state not included in the export (which would
indicate a divergence between Go EVM and revm), the guest panics with a
missing-state error. This triggers a proof failure, which the overlay
node handles by retrying (see "EVM Disagreement" in the overlay spec).

### State Export Serialization Format

The state export is RLP-encoded with the following structure:

```
StateExport = RLP([
    preStateRoot: bytes32,
    accounts: [AccountExport, ...],
    storageSlots: [StorageExport, ...],
    codeChunks: [CodeExport, ...]
])

AccountExport = RLP([
    address: bytes20,
    nonce: uint64,
    balance: uint256 (big-endian, no leading zeros),
    codeHash: bytes32,
    storageRoot: bytes32,
    proof: [bytes, ...]  // MPT proof nodes, RLP-encoded
])

StorageExport = RLP([
    address: bytes20,
    slot: bytes32,
    value: bytes32,
    proof: [bytes, ...]  // MPT proof nodes from storage trie
])

CodeExport = RLP([
    codeHash: bytes32,
    code: bytes
])
```

**Ordering**: Accounts are sorted by address (lexicographic). Storage slots are sorted by (address, slot). Code chunks are sorted by codeHash. Deterministic ordering ensures identical exports produce identical RLP encodings.

**Size limits**: Maximum state export size is 256 MB (configurable). If a batch touches more state than this limit, the batch must be split into smaller batches. In practice, a batch of 128 transactions touches ~1,000 storage slots, producing a ~2 MB export.

**Compression**: The export is zstd-compressed before passing to the SP1 prover as stdin. The guest decompresses on load.

### Proof Public Values Layout

The SP1 proof commits the following public values at fixed offsets:

| Offset | Size | Field | Description |
|--------|------|-------|-------------|
| 0 | 32 | preStateRoot | State root before batch execution |
| 32 | 32 | postStateRoot | State root after batch execution |
| 64 | 32 | receiptsHash | `keccak256(RLP(receipts))` |
| 96 | 8 | gasUsed | Total gas used in batch (uint64 BE) |
| 104 | 32 | batchDataHash | `hash256(batchData)` — double-SHA256 of the canonical batch encoding (see Spec 11, "Canonical Batch Data Encoding Format"). Uses hash256 (not keccak256) so the covenant can verify it with a single native `OP_HASH256`. |
| 136 | 8 | chainID | L2 chain ID (uint64 BE) — cross-shard replay prevention |
| 144 | 32 | withdrawalRoot | SHA256 Merkle root of withdrawal hashes in batch, or `bytes32(0)` if no withdrawals |
| 176 | 32 | inboxRootBefore | Inbox hash chain root before this batch (from host input). `bytes32(0)` if inbox is empty or inbox is disabled. See Spec 10, "Inbox Integration via STARK Public Values". |
| 208 | 32 | inboxRootAfter | Inbox hash chain root after this batch. `bytes32(0)` if all inbox txs were included (queue drained). Equals `inboxRootBefore` if no inbox txs were included in this batch. |
| 240 | 32 | migrateScriptHash | `SHA256(newScript)` if migration, else `bytes32(0)` |

Total: 272 bytes of public values.

The covenant extracts these values at the fixed offsets shown above. Any change to this layout requires a covenant migration.

**Receipt hash binding**: The `receiptsHash` in public values is committed by the guest but NOT verified by the covenant. It is verified by overlay nodes during batch replay — if a node computes a different receipts hash, it flags an EVM disagreement. The covenant verifies `preStateRoot`, `postStateRoot`, `batchDataHash`, `chainID`, and `migrateScriptHash`. This keeps the covenant script minimal while closing all data binding gaps.

**batchDataHash binding**: The `batchDataHash` uses `hash256` (double-SHA256, native `OP_HASH256`) — NOT `keccak256`. This is a deliberate choice: the covenant must independently verify that the batch data in the OP_RETURN matches what the STARK proof was generated for. Since Bitcoin Script has native `OP_HASH256` but no keccak256 opcode, the guest commits `hash256(batchData)` and the covenant verifies `hash256(batchData_from_unlocking_script) == batchDataHash_from_proof`. Without this check, a malicious prover could post a valid STARK proof (generated for batch B1) alongside different batch data B2 in the OP_RETURN — the state roots would be correct for B1 but nodes replaying from BSV data would execute B2 and compute wrong state roots, permanently splitting the shard. The `hashOutputs` check alone is insufficient because it only verifies the OP_RETURN contains the unlocking script's `batchData` parameter — it does not verify that parameter matches what the proof covers.

**Chain ID**: The `chainID` at offset 136 (8 bytes, big-endian uint64) is committed by the guest program and verified by the covenant's `advanceState` method against the shard's chain ID (a compile-time covenant property). This prevents cross-shard proof replay — see "Cross-shard proof replay prevention" section below.

**Withdrawal root**: The `withdrawalRoot` at offset 144 is committed by the SP1 guest and included in the OP_RETURN batch data. The state covenant does NOT verify it directly — it is covered by the `batchDataHash` binding (the batch data includes the withdrawal root, and the batch data hash is verified by the covenant). The bridge covenant reads the withdrawal root from the OP_RETURN of a confirmed covenant-advance tx and verifies individual withdrawals via SHA256 Merkle inclusion proofs. This avoids any Keccak-256 or Ethereum MPT verification in Bitcoin Script.

**Withdrawal Merkle tree construction**: The SP1 guest builds a binary SHA256 Merkle tree of all withdrawal hashes in the batch. Leaf nodes are the `hash256(bsvAddress || amount_u64_be || nonce_u64_be)` values read from the bridge contract's `withdrawalHashes` mapping. Internal nodes are `SHA256(left || right)`. If the number of leaves is not a power of 2, the tree is padded with `bytes32(0)` leaves. If there are no withdrawals, `withdrawalRoot = bytes32(0)`. The Go overlay node builds the same tree for the OP_RETURN encoding and for generating Merkle inclusion proofs when processing withdrawal claims.

## Component 3: STARK Verifier in BSV Covenant (Rúnar)

The covenant's locking script verifies the SP1 STARK proof. SP1 uses
FRI (Fast Reed-Solomon Interactive Oracle Proofs) for its STARK system.
The FRI verification algorithm requires:

- Hash operations (for Merkle tree commitments in FRI)
- Field arithmetic (additions, multiplications, inversions over a prime field)
- Polynomial evaluation checks

All of these are available in BSV Script via Rúnar. No elliptic curve
operations are required for STARK verification — this is purely hash-based
and arithmetic-based. This is simpler and cheaper than the Bulletproofs/
Groth16 verifiers considered in earlier spec versions.

### SP1 Proof Format

SP1 proofs have a well-defined structure:

```
SP1 STARK Proof:
  - FRI commitments (Merkle roots): N × 32 bytes
  - FRI query responses: Q queries × D layers × field elements
  - Public values: committed outputs from the guest program
  - Auxiliary data: permutation challenges, opening values
```

Typical proof size: 50-200KB (depending on circuit size and security
parameter). On BSV, this goes in the unlocking script of the covenant-
advance transaction.

### Covenant Contract

```go
// pkg/covenant/contracts/rollup.go

// GovernanceConfig determines the shard's trust model. Set at genesis,
// embedded in the covenant script as compile-time properties.
type GovernanceConfig struct {
    Mode      GovernanceMode
    Keys      [][]byte // Compressed public keys (for SingleKey or MultiSig)
    Threshold int      // M-of-N threshold (for MultiSig only)
}

type GovernanceMode int
const (
    GovernanceNone      GovernanceMode = iota // Fully trustless. No freeze, no upgrade.
    GovernanceSingleKey                       // Single key can freeze/unfreeze/upgrade.
    GovernanceMultiSig                        // M-of-N multisig can freeze/unfreeze/upgrade.
)

func RollupContract(sp1VK []byte, chainID uint64, gov GovernanceConfig) *runar.Contract {
    c := runar.NewStatefulContract("Rollup")

    // State: L2 state root, block number, inbox tracking, frozen flag
    c.State("stateRoot",          runar.Bytes32)
    c.State("blockNumber",        runar.Uint64)
    c.State("advancesSinceInbox", runar.Uint64) // Forced inclusion counter (see Spec 10)
    c.State("frozen",             runar.Uint8)  // 0 = active, 1 = frozen (governance)

    // SP1 verifying key (embedded in locking script)
    // This is derived from the guest program ELF — it uniquely
    // identifies WHICH program was proven.
    c.Prop("sp1VK", runar.VarBytes, sp1VerifyingKey)

    // Chain ID (embedded in locking script at genesis)
    // Cross-shard replay prevention: the covenant independently
    // verifies that the STARK proof was generated for THIS shard's
    // chain ID, even though the VK is already shard-specific.
    // See "Cross-shard proof replay prevention" section below.
    c.Prop("chainID", runar.Uint64, chainID)

    c.Method("advanceState", func(m *runar.MethodBuilder) {
        newStateRoot := m.Param("newStateRoot", runar.Bytes32)
        newBlockNum  := m.Param("newBlockNumber", runar.Uint64)
        proof        := m.Param("proof", runar.VarBytes)
        publicValues := m.Param("publicValues", runar.VarBytes)

        // 0. Reject if shard is frozen by governance
        m.Require(
            m.Equal(c.GetState("frozen"), runar.Uint8Literal(0)),
            "shard is frozen — advanceState blocked by governance",
        )

        // 1. Block number must be exactly previous + 1.
        //    Each covenant advance corresponds to exactly one L2 block.
        //    Proof aggregation (multiple blocks per advance) is a post-v1
        //    feature that would require a covenant migration to relax this
        //    check. Strict +1 is simpler and easier to verify.
        m.Require(
            m.Equal(newBlockNum, m.Add(c.GetState("blockNumber"), runar.Uint64Literal(1))),
            "block number must be exactly previous + 1",
        )

        // 2. Verify the SP1 STARK proof
        //    This checks: the proof is valid for the given verifying key
        //    and public values. The verifying key ensures the proof was
        //    generated by the specific guest program (our revm EVM).
        m.Require(
            runar.SP1Verify(proof, publicValues, c.Prop("sp1VK")),
            "invalid SP1 STARK proof",
        )

        // 3. Extract and verify public values
        //    Public values layout (offsets in bytes):
        //      [0..32]    preStateRoot
        //      [32..64]   postStateRoot
        //      [64..96]   receiptsHash
        //      [96..104]  gasUsed (uint64, big-endian)
        //      [104..136] batchDataHash = hash256(batchData) [verified by covenant]
        //      [136..144] chainID (uint64, big-endian) [verified by covenant]
        //      [144..176] withdrawalRoot (covered by batchDataHash binding)
        //      [176..208] inboxRootBefore (see inbox integration, Spec 10)
        //      [208..240] inboxRootAfter (see inbox integration, Spec 10)
        //      [240..272] migrateScriptHash (checked by migrate method)
        //    See "Proof Public Values Layout" section for definitive table.
        preRoot       := m.ExtractBytes32(publicValues, 0)
        postRoot      := m.ExtractBytes32(publicValues, 32)
        batchDataHash := m.ExtractBytes32(publicValues, 104)
        committedChainID := m.ExtractUint64(publicValues, 136)
        // withdrawalRoot at offset 144 is not verified by the state
        // covenant — it is covered by batchDataHash (the batch data
        // includes the withdrawal root). The bridge covenant reads it
        // from the OP_RETURN batch data of the referenced advance tx.

        // Inbox roots for forced inclusion (see Spec 10, "Inbox Integration")
        inboxRootBefore := m.ExtractBytes32(publicValues, 176)
        inboxRootAfter  := m.ExtractBytes32(publicValues, 208)

        // 4a. Chain ID must match this shard's chain ID (cross-shard replay prevention)
        m.Require(
            m.Equal(committedChainID, c.Prop("chainID")),
            "chain ID mismatch — cross-shard proof replay rejected",
        )

        // 4b. Pre-state root must match current covenant state
        m.Require(
            m.Equal(preRoot, c.GetState("stateRoot")),
            "pre-state root mismatch",
        )

        // 5. Post-state root must match claimed new state
        m.Require(
            m.Equal(postRoot, newStateRoot),
            "post-state root mismatch",
        )

        // 6. Verify batch data hash — two-layer binding.
        //
        //    Layer 1 (STARK → batch data): The STARK proof commits
        //    batchDataHash = hash256(batchData). The covenant independently
        //    computes hash256 of the batchData parameter from the unlocking
        //    script and verifies it matches batchDataHash from the proof.
        //    This uses native OP_HASH256 (one opcode, ~3 cycles).
        //
        //    Layer 2 (batch data → on-chain OP_RETURN): The covenant
        //    reconstructs the expected hashOutputs from the known outputs
        //    (covenant UTXO + OP_RETURN with batchData + fee change) and
        //    compares against the hashOutputs in the sighash preimage
        //    (via OP_PUSH_TX). This proves the OP_RETURN in the actual
        //    BSV transaction contains exactly the batchData from the
        //    unlocking script.
        //
        //    Together: proof covers batchData (layer 1), and batchData is
        //    in the OP_RETURN (layer 2). A malicious prover cannot post a
        //    valid proof for batch B1 alongside different data B2 — layer 1
        //    would reject because hash256(B2) != hash256(B1).

        batchData       := m.Param("batchData", runar.VarBytes)
        changeScript    := m.Param("changeScript", runar.VarBytes)
        changeSats      := m.Param("changeSats", runar.Uint64)

        // Layer 1: Verify the batch data matches what the STARK proof covers.
        // The guest committed: batchDataHash = hash256(batchData)
        // The covenant computes hash256(batchData) via native OP_HASH256
        // and compares against batchDataHash extracted from the proof's
        // public values at offset 104.
        m.Require(
            m.Equal(m.Hash256(batchData), batchDataHash),
            "batch data does not match proof — hash256(batchData) != committed batchDataHash",
        )

        // Layer 2: Verify the OP_RETURN output contains this exact batchData.
        // Extract hashOutputs from the sighash preimage (OP_PUSH_TX).
        hashOutputs := m.SighashField("hashOutputs") // 32 bytes from preimage

        // Reconstruct expected hashOutputs from the enforced outputs:
        //   Output 0: covenant UTXO (self-referencing)
        //   Output 1: OP_RETURN with "BSVM\x02" prefix + batch data
        //   Output 2: prover's fee change
        covenantOutput := m.SerialiseOutput(covenantSats, c.SelfScript())
        opReturnScript := m.Cat([]byte{0x00, 0x6a}, m.PushData(m.Cat([]byte("BSVM\x02"), batchData)))
        opReturnOutput := m.SerialiseOutput(0, opReturnScript)
        changeOutput   := m.SerialiseOutput(changeSats, changeScript)

        expectedHashOutputs := m.Hash256(m.Cat(covenantOutput, opReturnOutput, changeOutput))

        m.Require(
            m.Equal(hashOutputs, expectedHashOutputs),
            "hashOutputs mismatch — OP_RETURN does not contain the proven batch data",
        )

        // 7. Inbox forced inclusion (see Spec 10, "Inbox Integration via STARK Public Values")
        //    The SP1 guest commits inboxRootBefore and inboxRootAfter.
        //    The covenant enforces: if the inbox has been non-empty for
        //    too many advances, the batch MUST drain it.
        inboxIncluded := m.Not(m.Equal(inboxRootBefore, inboxRootAfter))
        newAdvanceCount := m.Add(c.GetState("advancesSinceInbox"), runar.Uint64Literal(1))

        m.Require(
            m.Or(
                m.LessThan(newAdvanceCount, runar.Uint64Literal(10)), // Not overdue
                m.Equal(inboxRootBefore, runar.Bytes32Zero),          // Inbox empty
                m.Equal(inboxRootAfter, runar.Bytes32Zero),           // All inbox txs included
            ),
            "forced inclusion: inbox txs must be fully included within 10 advances",
        )

        m.IfElse(inboxIncluded,
            func() { c.SetState("advancesSinceInbox", runar.Uint64Literal(0)) },
            func() { c.SetState("advancesSinceInbox", newAdvanceCount) },
        )

        // 8. Update state (preserve frozen=0 — advanceState only runs when active)
        c.SetState("stateRoot", newStateRoot)
        c.SetState("blockNumber", newBlockNum)
        c.SetState("frozen", runar.Uint8Literal(0))

        // 9. Enforce covenant recursion (output 0 is the new covenant UTXO)
        m.RequireStateOutput()
    })

    // --- Governance methods (conditional on genesis config) ---
    //
    // Governance state machine:
    //   ACTIVE → freeze()   → FROZEN
    //   FROZEN → unfreeze() → ACTIVE
    //   FROZEN → upgrade()  → FROZEN (new script)
    //   ACTIVE → upgrade()  → NOT ALLOWED (must freeze first)
    //
    // Governance modes:
    //   GovernanceNone:      No freeze/unfreeze/upgrade. Proof is sole authority.
    //   GovernanceSingleKey: Single key controls freeze/unfreeze/upgrade.
    //   GovernanceMultiSig:  M-of-N multisig controls freeze/unfreeze/upgrade.
    //
    // Governance keys CANNOT advance state — only a valid STARK proof can.
    // Governance keys CANNOT directly access bridge funds.
    // Governance keys CAN freeze the shard (pausing state transitions) and
    // upgrade the covenant script (only while frozen).

    switch gov.Mode {
    case GovernanceNone:
        // Fully trustless. No freeze, no unfreeze, no upgrade.
        // The only upgrade path is the proof-authorized migrate method
        // (spec 09, Milestone 10), which requires a valid STARK proof.

    case GovernanceSingleKey:
        c.Prop("governanceKey", runar.PubKey, gov.Keys[0])

        c.Method("freeze", func(m *runar.MethodBuilder) {
            sig := m.Param("sig", runar.Sig)
            m.RequireCheckSig(sig, c.Prop("governanceKey"))

            m.Require(
                m.Equal(c.GetState("frozen"), runar.Uint8Literal(0)),
                "already frozen",
            )

            c.SetState("frozen", runar.Uint8Literal(1))
            // stateRoot, blockNumber, advancesSinceInbox carried forward
            m.RequireStateOutput()
        })

        c.Method("unfreeze", func(m *runar.MethodBuilder) {
            sig := m.Param("sig", runar.Sig)
            m.RequireCheckSig(sig, c.Prop("governanceKey"))

            m.Require(
                m.Equal(c.GetState("frozen"), runar.Uint8Literal(1)),
                "not frozen",
            )

            c.SetState("frozen", runar.Uint8Literal(0))
            m.RequireStateOutput()
        })

        c.Method("upgrade", func(m *runar.MethodBuilder) {
            sig := m.Param("sig", runar.Sig)
            newScript := m.Param("newCovenantScript", runar.VarBytes)
            m.RequireCheckSig(sig, c.Prop("governanceKey"))

            m.Require(
                m.Equal(c.GetState("frozen"), runar.Uint8Literal(1)),
                "must freeze before upgrading",
            )

            // Replace the covenant script. State is NOT carried forward
            // via RequireStateOutput because the new script may have a
            // different state layout. The new covenant starts with whatever
            // state the new script's first output encodes.
            m.RequireOutput(0, newScript, m.InputValue(0))
        })

    case GovernanceMultiSig:
        c.Prop("governanceKeys", runar.PubKeyArray, gov.Keys)
        c.Prop("governanceThreshold", runar.Uint32, uint32(gov.Threshold))

        c.Method("freeze", func(m *runar.MethodBuilder) {
            sigs := m.Param("sigs", runar.SigArray)
            m.RequireMultiSig(sigs, c.Prop("governanceKeys"), c.Prop("governanceThreshold"))

            m.Require(
                m.Equal(c.GetState("frozen"), runar.Uint8Literal(0)),
                "already frozen",
            )

            c.SetState("frozen", runar.Uint8Literal(1))
            m.RequireStateOutput()
        })

        c.Method("unfreeze", func(m *runar.MethodBuilder) {
            sigs := m.Param("sigs", runar.SigArray)
            m.RequireMultiSig(sigs, c.Prop("governanceKeys"), c.Prop("governanceThreshold"))

            m.Require(
                m.Equal(c.GetState("frozen"), runar.Uint8Literal(1)),
                "not frozen",
            )

            c.SetState("frozen", runar.Uint8Literal(0))
            m.RequireStateOutput()
        })

        c.Method("upgrade", func(m *runar.MethodBuilder) {
            sigs := m.Param("sigs", runar.SigArray)
            newScript := m.Param("newCovenantScript", runar.VarBytes)
            m.RequireMultiSig(sigs, c.Prop("governanceKeys"), c.Prop("governanceThreshold"))

            m.Require(
                m.Equal(c.GetState("frozen"), runar.Uint8Literal(1)),
                "must freeze before upgrading",
            )

            m.RequireOutput(0, newScript, m.InputValue(0))
        })
    }

    return c.Build()
}
```

### Cross-shard proof replay prevention

A proof generated for Shard A must not be accepted by Shard B's
covenant. Three independent mechanisms prevent cross-shard replay:

1. **Shard-specific verifying key (VK)**: The chain ID is hardcoded in
   the SP1 guest program ELF (as `const CHAIN_ID: u64 = ...`). Since
   different shards have different chain IDs, their guest ELF binaries
   differ, and the SP1 verifying key (which is derived deterministically
   from the ELF) differs. A proof generated under VK_A will fail
   verification against VK_B embedded in Shard B's covenant. This is the
   primary defence.

2. **Committed chain ID in public values**: The guest program commits
   `chain_id` as a public value at offset 136 (8 bytes, big-endian
   uint64). The covenant's `advanceState` method extracts this value and
   verifies it matches the `chainID` property embedded in the covenant
   script at genesis. Even if an attacker found a VK collision (which
   would require breaking the hash function), the chain ID check
   independently rejects the proof.

3. **Pre-state root binding**: The proof commits to the pre-state root,
   which the covenant verifies matches its current state. Different
   shards have different state roots — a proof from Shard A referencing
   Shard A's state root would fail the pre-state check on Shard B.

These three checks are independent — any one of them is sufficient to
prevent cross-shard replay. Together they provide defence in depth.

**Guest program chain ID hardcoding**: The chain ID is a compile-time
constant in the guest program, not a runtime input:

```rust
// prover/guest/src/main.rs
const CHAIN_ID: u64 = 8453111; // Set at build time per shard
```

This means each shard has its own guest ELF binary and its own VK. The
`RollupContract(sp1VK, chainID, governanceConfig)` function receives
these values at genesis and embeds them as compile-time properties.
The covenant script differs per shard (due to VK, chain ID, and
governance configuration).

### The SP1Verify Subroutine in Rúnar

The `runar.SP1Verify()` function compiles to Bitcoin Script that implements
the FRI verification algorithm. This is the critical piece that must work
on BSV.

FRI verification steps (all hash + arithmetic, no EC):

```
1. Deserialise proof components (Merkle roots, query responses, etc.)
2. Recompute Fiat-Shamir challenges from the proof transcript (hash chain)
3. For each FRI round:
   a. Verify Merkle proof of the queried position
   b. Check the folding equation: f(x²) = (f(x) + f(-x))/2 + α·(f(x) - f(-x))/(2x)
   c. This is field arithmetic: additions, multiplications, inversions
4. Check the final polynomial evaluation
5. Verify the DEEP-ALI composition (polynomial identity check)
```

All operations are:
- `hash256` (SHA-256 double-hash, native BSV opcode)
- Field arithmetic over the Baby Bear field (p = 2^31 - 2^27 + 1)
  using BSV arithmetic opcodes (OP_ADD, OP_MUL, OP_MOD, etc.)
- Merkle proof verification (repeated hashing)

### Script Size and Execution Cost Estimate

FRI verification complexity:
- Security parameter λ: 100 bits
- Number of FRI queries: ~30 (for 100-bit security)
- FRI rounds: ~20 (log of the trace length)
- Per query: ~20 hash operations + ~50 field multiplications
- Total: ~600 hashes + ~1,500 field multiplications + Merkle checks

Estimated script size: **200KB - 2MB** depending on optimisation
(inlining vs looping, precomputed tables).

BSV limits: 4GB script, unlimited execution time for miners.

This MUST be validated as Gate 0 (see spec 09, Milestone 0) before
any implementation proceeds. The targets are:

- Script size: < 10MB
- Stack depth: < 1,000 elements
- Execution time: < 1 second on BSV regtest

## Component 4: Overlay Node Integration

The overlay node gains a proving step between execution and broadcast.
Execution and proving are decoupled — execution runs ahead, proving
catches up asynchronously.

```go
// pkg/overlay/process.go (updated for SP1)

func (n *OverlayNode) ProcessBatch(txs []*Transaction) ([]*Receipt, error) {
    n.mu.Lock()
    defer n.mu.Unlock()

    // Reject if shard is frozen by governance
    if n.governanceState == GovernanceFrozen {
        return nil, fmt.Errorf("shard is frozen by governance — no state transitions allowed")
    }

    // === FAST PATH: Go EVM execution (instant) ===

    parentHeader := n.chainDB.ReadHeadHeader()
    statedb, _ := state.New(parentHeader.StateRoot, n.db)

    // Start cumulative access recording BEFORE execution.
    statedb.StartAccessRecording()

    // Execute all transactions through Go EVM
    var receipts []*Receipt
    var gasUsed uint64
    for _, tx := range txs {
        receipt, _ := ApplyTransaction(evm, statedb, header, tx, &gasUsed)
        receipts = append(receipts, receipt)
    }

    stateRoot := statedb.IntermediateRoot(true)

    // Stop recording and capture the cumulative access set
    accessRecording := statedb.StopAccessRecording()

    // Export the accessed state for the SP1 guest BEFORE commit.
    // CRITICAL: We must generate Merkle proofs against the PRE-STATE
    // trie (the SP1 guest verifies it starts from the correct root).
    // Open a fresh read-only statedb at the parent's state root.
    preStateDB, _ := state.New(parentHeader.StateRoot, n.db)
    stateExport, err := ExportStateForProving(
        preStateDB,                      // read-only, at pre-state root
        accessRecording.Accounts,        // all accounts accessed
        accessRecording.Slots,           // all slots accessed per account
    )
    if err != nil {
        return nil, fmt.Errorf("state export failed: %w", err)
    }

    // Commit state locally
    statedb.Commit(true)

    // Store L2 block
    l2Block := buildL2Block(header, txs, receipts, stateRoot)
    n.chainDB.WriteBlock(l2Block)

    // === SLOW PATH: SP1 proving (async) ===

    // Queue proving job
    n.provingQueue <- &ProvingJob{
        L2Block:     l2Block,
        PreState:    parentHeader.StateRoot,
        StateExport: stateExport,
        Transactions: encodeTxBatch(txs),
        BlockContext: header.ToBlockContext(),
        Callback:    n.onProofComplete,
    }

    // Return receipts immediately — don't wait for proof
    return receipts, nil
}

func (n *OverlayNode) onProofComplete(job *ProvingJob, output *ProveOutput, err error) {
    if err != nil {
        log.Error("Proving failed", "block", job.L2Block.Number, "err", err)
        // Retry or alert — the shard stalls until a proof is produced
        return
    }

    // Verify the SP1 output matches the Go EVM output.
    // If they disagree, do NOT broadcast the BSV transaction — the
    // covenant chain is never corrupted because the mismatched proof
    // never reaches BSV. Re-queue for proving retry.
    //
    // In a multi-node shard, another node will likely advance the
    // covenant independently. This node detects the advance via the
    // double-spend monitor and replays the winner's batch.
    //
    // The receipt already returned to the user was speculative (pre-
    // confirmation), same as any L2 receipt before L1 finality.
    if output.PostStateRoot != job.L2Block.Header.StateRoot {
        log.Error("EVM DISAGREEMENT — retrying proof",
            "goRoot", job.L2Block.Header.StateRoot,
            "sp1Root", output.PostStateRoot,
        )
        // Re-queue for retry. If disagreement persists across retries,
        // the node operator must investigate (likely a bug in one EVM).
        pp.queue <- job
        return
    }

    // Build BSV covenant-advance transaction with the proof
    bsvTx, _ := n.buildCovenantAdvanceTx(job.L2Block, output.Proof, output.PublicValues)

    // Broadcast to BSV
    bsvTxID, _ := n.bsvClient.Broadcast(bsvTx)

    // Cache
    n.txCache.Append(&CachedTx{
        BSVTx:      bsvTx,
        BSVTxID:    bsvTxID,
        L2Block:    job.L2Block,
        StateRoot:  output.PostStateRoot,
        L2BlockNum: job.L2Block.Header.Number.Uint64(),
    })
}
```

### Execution Tip vs Proven Tip

The overlay node maintains two chain tips:

- **Execution tip**: The latest state after executing all received
  transactions through the Go EVM. User-facing. Receipts are returned
  from this tip. Advances instantly.

- **Proven tip**: The latest state with a completed SP1 proof that has
  been broadcast to BSV. The covenant chain follows this tip. Advances
  as proofs complete (seconds behind execution tip).

```go
type ChainTips struct {
    ExecutionTip uint64 // Latest executed L2 block
    ProvenTip    uint64 // Latest proved + broadcast L2 block
}
```

The RPC maps block tags accordingly (see spec 05 for authoritative definitions):
- `"latest"` → execution tip (fast, may be unproven)
- `"safe"` → proven tip (SP1 proof generated + BSV broadcast)
- `"finalized"` → proven tip with ≥6 BSV confirmations

### Parallel Proving

Multiple proving jobs can run concurrently:

```go
// pkg/prover/parallel.go

type ParallelProver struct {
    sp1     *SP1Prover
    workers int
    queue   chan *ProvingJob
}

func (pp *ParallelProver) Run(ctx context.Context) {
    for i := 0; i < pp.workers; i++ {
        go func() {
            for job := range pp.queue {
                output, err := pp.sp1.Prove(&ProveInput{
                    PreStateRoot:    job.PreState,
                    SerialisedState: job.StateExport,
                    Transactions:    job.Transactions,
                    BlockContext:    job.BlockContext,
                })
                job.Callback(job, output, err)
            }
        }()
    }
}
```

Proving jobs are independent (each batch references only its pre-state
and transactions). However, the BSV covenant chain is sequential — batch
N's proof must be broadcast before batch N+1's proof can spend the
covenant UTXO. The overlay node queues covenant advances in order even
if proofs complete out of order.

### Head-of-Line Blocking Mitigation

If block N is slow to prove (complex DeFi batch) while block N+1 is
fast (simple transfers), N+1's proof waits for N. To prevent unbounded
latency:

1. **Proving timeout**: If a proving job exceeds `ProvingTimeout`
   (default: 60s), it is cancelled and the block is merged with the
   next pending block into a combined proving job. The combined job
   proves the composite state transition N-1 → N+1 in a single proof.

   **Covenant implication**: The v1 covenant requires `newBlockNumber ==
   blockNumber + 1` (strict +1). Merged proofs that skip block numbers
   are NOT supported in v1. If block N times out, it is retried or the
   node enters follower mode — it is NOT merged with N+1. Proof
   aggregation (multiple blocks per advance) is a post-v1 feature that
   would require a covenant migration to relax the block number check.

2. **Priority ordering**: The proving queue processes jobs in block
   number order. If block N is still proving when N+1 and N+2 complete,
   the node holds N+1 and N+2 until N finishes or times out.

3. **Proving stall detection**: If no proof completes within
   `3 × ProvingTimeout`, the node enters follower-only mode, fires
   an operator alert, and waits for another node to advance the
   covenant.

```go
type ProvingConfig struct {
    Workers        int           // Parallel proving goroutines
    ProvingTimeout time.Duration // Max time per proof (default: 60s)
    StallTimeout   time.Duration // Enter follower mode after this (default: 180s)
}
```

## Proving Scalability

### Horizontal Scaling via Shard Parallelism

SP1 proving is horizontally scalable. The SP1 prover breaks the RISC-V
execution trace into shards (segments of ~2^21 cycles each). Each shard
is proven independently and in parallel. The shard proofs are then
recursively combined into a single proof. A 128-tx batch at ~60M cycles
produces roughly 30 shards.

Proving time scales inversely with GPU count:

| Hardware | Parallelism | 128-tx proving time | Sustained throughput |
|---|---|---|---|
| 1 GPU (A100) | Sequential shards | ~5-10s | ~13-25 tx/sec |
| 4 GPUs | 4 shards in parallel | ~2-3s | ~40-60 tx/sec |
| 8 GPUs | 8 shards in parallel | ~1-2s | ~60-120 tx/sec |
| 32 GPUs (cluster) | All shards in parallel | ~0.5-1s | ~120-250 tx/sec |
| SP1 Prover Network | Elastic, hundreds of GPUs | sub-second | 500+ tx/sec |

The recursive combination step adds a fixed ~1-2 seconds regardless of
shard count. With sufficient GPUs, the per-shard phase completes in one
shard's time.

Proving time is not a fixed architectural constraint — it is an
operational variable that scales with hardware investment. A stablecoin
shard doing 10 tx/sec needs one modest GPU. A high-volume DeFi shard
doing 500 tx/sec needs a GPU cluster or SP1 Prover Network access.

### Prover Deployment Options

Three deployment models are supported:

1. **Self-hosted GPU**: Operator runs their own GPU(s). Full control,
   fixed cost. Throughput scales linearly with GPU count.
2. **SP1 Prover Network**: Succinct Labs' decentralized prover network.
   The overlay node submits proving requests over the network and
   receives proofs back. No local GPU needed. Pay-per-proof. Elastic —
   handles traffic spikes without pre-provisioned hardware.
3. **Hybrid**: Self-hosted for baseline load, prover network for spikes.

```go
type SP1ProverConfig struct {
    GuestELFPath          string        // Path to compiled guest ELF
    ProverBinary          string        // Path to SP1 prover binary (local mode)
    Timeout               time.Duration // Max proving time before abort
    Workers               int           // Parallel proving workers
    GPUCount              int           // Number of local GPUs for parallel shard proving
    ProverNetworkURL      string        // SP1 Prover Network endpoint (optional)
    ProverPoolURL         string        // Custom proving pool endpoint (optional)
    ProverNetworkFallback bool          // Use network/pool as fallback if local proving too slow
}
```

### Practical Throughput Ceiling

With sub-second proving, the bottleneck shifts to BSV transaction
propagation (~200-500ms per transaction). The practical ceiling is
roughly one covenant advance per 200-500ms, or ~2-5 advances per
second. At 128 txs per batch, that is ~250-640 tx/sec per shard — far
beyond foreseeable demand for a single shard.

For workloads exceeding this ceiling, the multi-shard architecture
provides horizontal scaling: deploy additional shards, each with its
own covenant chain and prover infrastructure.

### Future: Proof Aggregation

SP1 supports recursive proof composition. Instead of one covenant
advance per batch, aggregate N batch proofs into a single recursive
proof and advance the covenant once for N batches. This reduces BSV
transactions by N× and amortises covenant overhead. Trade-off: higher
latency (wait for N batches) and more complex prover infrastructure.

The v1 covenant uses strict `Equal+1` for block number checks. Proof
aggregation requires relaxing this to `GreaterThan`, which is a covenant
migration (see spec 09, Milestone 10). The migration is straightforward
— change one comparison opcode — but requires the full migration
procedure (freeze, upgrade, unfreeze).

## BSV Covenant-Advance Transaction Layout

```
Input 0:  Previous covenant UTXO           [carries fixed sats]
Input 1:  Prover's funding UTXO            [pays BSV mining fee]

Output 0: New covenant UTXO                [same fixed sats]
          Locking script: Rúnar-compiled FRI verifier + state
          State data: [stateRoot, blockNumber]

Output 1: OP_RETURN batch data             [0 sats]
          "BSVM" 0x02 <version> <compression> <blockNum>
          <preStateRoot> <postStateRoot> <txCount>
          <RLP-encoded transactions, zstd-compressed>
          (~20KB for 128-tx batch)

Output 2: Prover's change                  [input_1 - mining_fee]

Unlocking script for Input 0:
  PUSH newStateRoot          [32 bytes]
  PUSH newBlockNumber        [8 bytes]
  PUSH sp1Proof              [~165KB for 128-tx batch]
  PUSH publicValues          [~128 bytes]
  PUSH sighashPreimage       [variable, for OP_PUSH_TX]

Total transaction size: ~216KB for a 128-tx batch.
BSV fee at 100 sats/KB: ~21,600 sats (~$0.0065).
Per EVM transaction L1 cost: ~169 sats (~$0.00005).
```

### Batch Data Encoding

See Spec 11, "Canonical Batch Data Encoding Format" for the definitive
wire format, field offsets, and hash domain definition. The key points
for the prover are:

- `batchDataHash = hash256(batchEncoding)` where hash256 is double-SHA256
  (`OP_HASH256`), NOT keccak256. The covenant verifies this natively.
- The hash covers everything after the 5-byte envelope header (`BSVM\x02`)
- The `withdrawalRoot` is at offset 173 in the canonical encoding
- v1 uses no compression — the encoding is stored uncompressed

Both the Go overlay node and the Rust SP1 guest program MUST produce
identical encodings for the same block.

## Security Model

### What the proof guarantees

The SP1 STARK proof guarantees:

1. **Full EVM execution correctness**: Every opcode executed correctly.
   A → B transfer means A's balance decreased and B's increased by
   exactly the right amount.

2. **State consistency**: The post-state root is the correct result of
   applying the transactions to the pre-state.

3. **Transaction integrity**: The transactions proven are the transactions
   in the batch. The prover cannot substitute different transactions.

4. **Gas accounting**: Gas was computed correctly for every operation.

5. **Determinism**: The same inputs always produce the same outputs. Two
   independent provers generating proofs for the same batch will commit
   the same public values (different proof bytes, but same verified outputs).

### What the proof does NOT guarantee

1. **Data availability**: The proof doesn't guarantee the batch data is
   published. The covenant enforces this by requiring the OP_RETURN output
   (Output 1) and by verifying the batch data hash from the proof's public
   values matches the OP_RETURN content. This cryptographic binding ensures
   the published batch data is exactly the data the guest program executed.
   Without this binding, a malicious prover could post a valid proof with
   garbage batch data, breaking replay verification by other nodes.

2. **Liveness**: The proof doesn't guarantee someone will generate proofs.
   If all provers go offline, the shard stalls. This is mitigated by the
   multi-node model (spec 11) — any node can prove.

3. **Transaction inclusion**: The proof doesn't guarantee which transactions
   are included. The prover chooses which transactions to batch. This is the
   same MEV surface as all L2s (the prover controls ordering and inclusion).

### Trust assumptions

1. **SP1 soundness**: The STARK proof system is sound. A forged proof
   cannot pass verification. This is a cryptographic assumption backed by
   mathematical proofs and multiple audits.

2. **revm correctness**: The Rust EVM correctly implements Ethereum
   semantics. This is validated by passing ethereum/tests.

3. **Rúnar compiler correctness**: The FRI verifier compiles correctly
   to Bitcoin Script. Mitigated by multi-implementation conformance
   (compile with Go + Rust + TS, verify byte-identical output).

4. **SP1 verifying key binding**: The verifying key in the covenant
   uniquely identifies the guest program. A different guest program
   (e.g., one with a backdoor) would have a different verifying key
   and would not pass the covenant's VK check.

**Cross-shard proof replay prevention**: Each shard has a unique SP1 verifying key (derived from its guest program ELF, which embeds the shard's chain ID). A proof generated for shard A cannot pass verification on shard B's covenant because the verifying keys differ. The chain ID is hardcoded in the guest program and included in EVM transaction signature verification (EIP-155), making the guest program — and therefore the VK — shard-specific.

### Governance Trust Model

The governance configuration determines the shard's trust assumptions
beyond the cryptographic guarantees of the STARK proof system:

**GovernanceNone**: Fully trustless. The shard's security depends
entirely on STARK soundness, compiler correctness, and the EVM
implementations. No party can freeze, upgrade, or intervene. If the
FRI verifier has a bug, the shard is permanently compromised. This is
the long-term target once the codebase is battle-tested and formally
verified. Recommended only for shards with mature, audited covenants.

**GovernanceSingleKey**: Maximum flexibility, maximum trust. A single
operator can freeze the shard (pausing all state transitions), upgrade
the covenant script, and unfreeze. Suitable for development, testing,
and single-operator shards where the operator is trusted.

**GovernanceMultiSig**: Distributed governance. M-of-N key holders
must cooperate for all governance operations. Recommended for
production shards holding significant value. Balances recoverability
(can freeze and patch a buggy covenant) with trust distribution (no
single party controls the shard).

**What governance keys CAN do**:
- Freeze the shard (block `advanceState`, pause all state transitions)
- Unfreeze the shard (resume normal operation)
- Upgrade the covenant script (only while frozen)
- Freeze the bridge covenant (block withdrawal claims)

**What governance keys CANNOT do**:
- Advance state (only a valid STARK proof can do this)
- Directly access bridge funds (bridge is proof-authorized)
- Operate on an active shard (must freeze first, which is visible
  to all observers on BSV)
- Alter historical state (the covenant UTXO chain is immutable)

The freeze-before-upgrade requirement ensures the community has time
to observe and react before a governance action takes effect. A freeze
is visible on BSV immediately — any observer can detect it by
monitoring the covenant UTXO chain. Users should evaluate the
governance model before depositing funds.

## Testing

### Correctness

1. **Cross-EVM agreement**: For every ethereum/tests fixture, run both
   the Go EVM and the SP1 guest (revm). Verify identical state roots.

### Cross-EVM Differential Testing

State root agreement between Go EVM (geth extraction) and Rust revm (SP1 guest) is the single most critical correctness property. The following test infrastructure is REQUIRED:

1. **ethereum/tests fixture runner**: Run the full `GeneralStateTests` and `BlockchainTests` suites against BOTH the Go EVM and a standalone revm binary. Compare state roots bit-for-bit after each test.

2. **Differential fuzzer**: Generate random EVM transactions (using go-fuzz or similar), execute against both implementations, compare:
   - Post-state root
   - Gas used
   - Return data
   - Log entries (topics + data)
   - Account nonces and balances

3. **MPT consistency test**: For every test that modifies storage, verify that the Go MPT (`pkg/mpt/`) and Rust MPT (`alloy-trie`) produce identical roots given identical account/storage state.

4. **Acceptance criterion**: Zero divergences across the full ethereum/tests suite. Any divergence is a P0 blocker.

5. **CI integration**: The differential tests run on every commit. A divergence fails the build.

2. **Proof round-trip**: Generate an SP1 proof for a known state
   transition. Verify the proof using SP1's native Rust verifier.
   Then verify the proof using the Rúnar-compiled FRI verifier on
   BSV regtest. All three must agree.

3. **Covenant integration**: Deploy the covenant to BSV regtest.
   Submit a covenant-advance with a valid SP1 proof. Verify it succeeds.
   Submit with an invalid proof. Verify BSV miners reject the tx.

### Soundness

4. **Tampered proof**: Flip bits in the proof → covenant must reject.
5. **Wrong state root**: Valid proof but mismatched state root in the
   covenant parameters → covenant must reject (pre-state root check).
6. **Wrong verifying key**: Proof from a different guest program →
   covenant must reject (VK mismatch).
7. **Replayed proof**: Reuse a valid proof from block N for block N+1 →
   covenant must reject (pre-state root mismatch).

### Performance

8. **Proving time**: Benchmark for batch sizes 1, 10, 50, 100.
   Target: < 30 seconds for 100-tx batch.
9. **Verification time**: Measure FRI verification on BSV regtest.
   Target: < 1 second.
10. **Script size**: Measure compiled covenant script.
    Target: < 10MB.
